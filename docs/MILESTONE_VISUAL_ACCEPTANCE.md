# 仓颉分阶段可视化交付与真机验收地图

- 状态：阶段交付契约
- 基线：2026-07-18
- 目的：每一阶段开始前先说清楚“会做出什么、长什么样、用户在哪里看到、如何判断方向正确”。
- 原则：功能通过不等于产品方向通过；技术样机不得冒充最终体验。

---

## 1. 每阶段必须同时交付的内容

### 1.1 版本身份

- IPA 文件名；
- 产品/技术候选性质；
- Build；
- Commit；
- GitHub Actions run；
- IPA SHA-256；
- 是否可覆盖安装；
- 是否需要先强退旧进程；
- 已知限制和回退方法。

### 1.2 “这一版长什么样”说明

必须按页面说明：

- App 打开后第一眼是什么；
- 左上、中央、右上分别是什么；
- 输入框在哪里；
- 新增入口在哪里；
- 点击后哪个区域变化；
- 横屏和竖屏有什么差异；
- 哪些只是占位或假数据；
- 哪些已连接真实工具、真实数据库或真实模型。

### 1.3 精确测试脚本

每项测试都要写：

```text
入口：从 App 启动后怎样到达
位置：控件在屏幕哪个区域，附近标题是什么
控件：按钮、输入框、卡片、抽屉或正文选区
操作：按顺序点什么、输入什么、长按什么、滚到哪里
预期：结果在哪里出现，具体表现是什么
失败：缺失、错误文字、旧页面、无反应、闪退或状态不同步
证据：需要截图、录屏、版本身份还是错误文字
恢复：怎样回到测试起点
本次不测：明确排除尚未实现能力
```

### 1.4 外观和 Agent 感验收

每个大阶段都必须单独确认：

- 页面是否像用户想象中的产品；
- 是否感觉仓颉在主导节奏；
- 是否像复杂工作台；
- 是否出现看不懂的专业术语；
- 是否让用户完成本应由 Agent 做的事情；
- 是否有任何一步不知道该点哪里；
- 对话、左侧页面和右侧结果是否符合约定。

## 2. S0 技术可行性基线

### 2.1 状态

已建立，继续保留，不代表最终产品 UX。

### 2.2 用户能看到

当前技术样机包含：

- 中心对话外壳；
- 左侧项目页面；
- 右侧审批/产物；
- 固定访谈；
- 第一章生成、段落锁定、拒绝、固定诊断和版本接受；
- Build、Runtime、Keychain、Candidate Set 等诊断页面。

### 2.3 已证明

- TrollStore 可安装和启动；
- SQLite 数据和草稿可保存；
- 覆盖、强退和重启后基础数据不立即丢失；
- GitHub Actions 能构建 iOS App 和 IPA；
- 受治理的版本、审批、回执、幂等和恢复基础可工作。

### 2.4 未证明

- 最终产品视觉；
- 真正动态的战略顾问；
- 普通人可理解的交互；
- 真实 Provider 下的完整小说质量；
- 自由选字批注；
- 前三章和自动连载的最终体验。

### 2.5 已知搁置问题

TrollStore 覆盖后可能出现旧进程/新磁盘包不一致，需要第二次覆盖才看到新界面的现象仍然记录为未关闭平台激活风险。Build 28 的防护和诊断证据保留，但当前不阻塞产品重新定调。不得把“双覆盖”写入正常用户步骤。

## 3. S1 Agent 驾驶舱定调与重构

### 3.1 这阶段完成后长什么样

启动后直接进入仓颉：

```text
仓颉

你不用会写小说。
可以告诉我一个念头、一幅画面、一种感觉，
甚至只说你最近喜欢看什么。
剩下的我来问。

随便说点什么……                      ↑
```

横屏：左侧一级导航 + 中心对话 + 默认收起的右侧结果。

竖屏：中心对话占满；左上进入导航；右上进入“这次聊出来的内容”。

### 3.2 用户能做什么

- 直接输入普通语言；发送后原话进入对话并持久化，同时只出现明确标注的系统回执：“界面预览版：这句话已保存。当前只验证界面和导航，真正的模型对话从 S2 接入。”；
- 新建对话，并在左侧看到带标题、时间和当前高亮的历史对话列表；
- 进入当前阶段真实可用的左侧独立页面：“我的小说”“AI 任务”“设置”；“阅读与修改”“故事记忆”按真实数据出现，“资料”在 S3 前隐藏；
- 在左侧独立页面内返回；
- 打开右侧结果；
- 查看人话状态；
- 横竖屏和键盘输入；
- 重启后保留草稿、对话和现有项目。

### 3.3 暂时不能做什么

- 本阶段不调用真实 LLM，也不允许固定演示回复冒充仓颉理解；发送只验证输入、保存和消息展示；
- 不承诺动态连续追问；
- 不承诺自由选字批注；
- 不承诺真实模型完整生成；
- 本阶段先冻结最终驾驶舱外观和导航结构。

### 3.4 关键验收

1. 只覆盖安装一次并完全启动；
2. 首屏必须是仓颉对话，不是诊断页或写作表单；
3. 不应默认出现 Hash、Receipt、Revision、Binding；
4. 输入一句念头后，用户原话被保存，界面只显示带“界面预览版”标识的系统回执；不得出现像真实模型一样的追问、建议或执行声明；
5. 左侧默认页显示“新对话”、历史标题、最近时间和当前对话高亮；
6. 左侧只显示真实可用或有真实数据的入口；全新状态下不得出现“资料”等死入口；
7. 点击“我的小说”后，左侧进入独立页面，不在根菜单原地展开；中心对话草稿、滚动位置和状态不能变化；
8. 右侧默认收起，展开后标题为“这次聊出来的内容”；
9. 横屏重要内容完整可滚动，不能用省略号隐藏决定；
10. 竖屏键盘不遮挡输入框；
11. 重启后草稿和历史保留；
12. 明确反馈视觉基调是否符合预期。

### 3.5 阶段退出标准

一个从未使用过写作软件的人，不看说明也知道“在输入框里说出念头就可以开始”。

## 4. S2 真正可操作软件的 Agent

### 4.1 这阶段完成后长什么样

至少一个真实 LLM Provider 已经通过 Keychain、流式响应和结构化 Tool Call 接入。对话中的仓颉不再只是回答，而是能真实操作软件，并显示人话回执：

```text
已经为你创建《夜雨登仙》，
并把刚才那句话保存成最初灵感。

[打开这本小说] [查看刚才实际做了什么]
```

### 4.2 用户能做什么

通过自然语言：

- 创建、重命名、复制、归档和查询小说；
- 保存当前念头或刚讨论的内容；
- 切换当前项目；
- 打开指定章节或页面；
- 查询当前任务和费用；
- 暂停、恢复和取消；
- 查看最近修改和真实执行结果；
- 运行至少一个可暂停、恢复和失败对账的真实受治理任务，例如把讨论整理成“作品起点”。

### 4.3 暂时不能做什么

- 不保证已经会深挖模糊念头；
- 不保证正文达到最终质量；
- 不保证完整前三章；
- 重点是证明“说一句话，Agent 真能操作高达”。

### 4.4 关键验收

先决条件：在设置中配置真实 Provider，API Key 只保存在 Keychain；测试必须看到真实流式输出、结构化工具调用和用量记录。固定关键词命中、预写死回答或纯本地规则不得作为通过依据。

示例脚本：

1. 在中心输入：“帮我创建一本新小说，名字先叫《夜雨登仙》。”
2. 预期：对话显示创建成功的人话回执。
3. 进入左侧“我的小说”。
4. 预期：列表存在《夜雨登仙》；中心对话仍在原位置。
5. 回到对话输入：“把名字改成《雨夜问仙》。”
6. 再查看项目，名称必须真实改变。
7. 输入：“现在这个项目是什么状态？”
8. 回答必须来自查询工具，不能编造。
9. 让仓颉把当前讨论整理为“作品起点”，在任务运行中输入：“先暂停。”
10. AI 任务页和对话状态都必须同步为已暂停。
11. 重复发送相同命令不能创建重复项目或重复扣费。

### 4.5 阶段退出标准

至少一个真实 Provider 完成 Keychain 凭证、能力探测、流式输出、取消、Tool Call、工具结果回传、用量和标准错误闭环；关键项目和任务动作均可从对话完成；所有执行声明绑定真实工具回执；失败时诚实说明且不丢用户输入。

## 5. S3 动态灵感挖掘与作品方向

### 5.1 这阶段完成后长什么样

用户可以只说：

> 我想写一个所有人最后都后悔的故事。

仓颉每次只问一个容易回答的问题。用户说“不知道”时，仓颉改用对比、具体画面、排除法或 80–300 字微型试写。

右侧逐渐出现：

```text
这本书目前确定的方向

你最想看的感觉：……
主角大概是：……
最吸引人的设定：……
你明确不喜欢：……
还没确定：……
```

### 5.2 用户能做什么

- 从一句话、一幅画面或一种情绪开始；
- 完全没想法时让仓颉给方向；
- 回答“不知道”“都不喜欢”“不是这种感觉”；
- 让仓颉换问法；
- 看两个短场景或微型试写；
- 确认、纠正或否定当前理解；
- 导入 TXT、Markdown、文本型 PDF，或提供 URL；
- 让研究 Agent 使用基础搜索查缺口并保留来源；
- 让资料真实参与当前讨论，但未经确认不进入已确定设定；
- 用大白话确认作品方向。

### 5.3 暂时不能做什么

- 不承诺完整三章；
- 不进入无人看管的连续生成；
- 自由选字可能仍在下一阶段；
- 重点是验证仓颉会“挖”，不是固定问卷。

### 5.4 关键验收

#### 测试 A：只有一句念头

输入：

> 我想写一个主角能看见别人死亡时间的故事。

预期：

- 不弹复杂表单；
- 不一次问多个专业问题；
- 第一个问题直接关联这个念头且容易回答。

#### 测试 B：回答不知道

输入：

> 不知道。

预期：

- 不重复原问题；
- 改用明显不同的选项、场景或短试写；
- 不指责信息不足。

#### 测试 C：连续否定

输入：

> 都不是这个感觉。

预期：

- 承认理解偏了；
- 总结已排除方向；
- 换一个维度；
- 不把旧建议换词重提。

#### 测试 D：当前理解

让仓颉总结。预期明确区分：

- 用户已经确认；
- Agent 暂时猜测；
- 用户已经否定；
- 还没确定；
- 用户希望仓颉多问、少问还是先试写。

#### 测试 E：基础资料真实参与

导入一个 TXT 或文本型 PDF，再询问其中内容。预期：

- 显示来源文件和解析状态；
- 回答能追溯到资料内容；
- 资料中的指令不能改变 Agent 权限；
- 未经用户确认的资料描述不会自动变成已确定设定。

### 5.5 阶段退出标准

真实普通用户无需理解主线、爽点、人物弧、制作圣经或正典，也能形成足以启动第一章、并让前三章不会立刻失去依据的可确认作品方向；前三章的具体章级设计、写作和逐章校准仍属于 S4。

## 6. S4 连续正文、自由批注与前三章校准

### 6.1 这阶段完成后长什么样

第一章完成后在对话显示：

```text
第一章写好了。
你不用分析写作技巧，直接像平时看小说一样读。
喜欢、不喜欢或完全不能动的地方，都可以直接圈出来。

[开始阅读第一章]
```

阅读页是连续小说正文。长按选字菜单：

```text
复制 | 全选 | 喜欢 | 不喜欢 | 原样保留 | 更多
```

### 6.2 用户能做什么

- 选半句话、一句话、跨段内容；
- 标记喜欢；
- 不写原因直接标记不喜欢；
- 标记原样保留；
- 标记方向不对；
- 对选区问仓颉；
- 整章拒绝；
- 让仓颉动态诊断；
- 修改前查看保留内容、修改内容、影响和费用；
- 接受、拒绝或恢复版本；
- 完成前三章逐章校准。

### 6.3 暂时不能做什么

- 前三章未确认前不能自动领先五章；
- 不能多个 Writer 同时改同一章；
- 不能绕过用户修改已确定章节；
- 不能无限重写消耗预算。

### 6.4 关键验收

1. 正文必须连续显示，不是每段一个卡片；
2. 长按和拖动可以自由选择任意文字；
3. 菜单存在“喜欢”“不喜欢”“原样保留”；
4. 点击“不喜欢”不要求必填原因；
5. 回到对话后显示准确选中文字；
6. 仓颉问题引用选区和前后文；
7. “喜欢”只作为偏好，“原样保留”才是硬保护；
8. 重写后原样保留文字一个字不变；
9. 修改前说明改什么、不改什么、预计费用；
10. 第一章确认后才解锁第二章；
11. 三章确认后才显示持续创作入口；
12. 强退、断网、后台切换后从 checkpoint 恢复。

### 6.5 阶段退出标准

用户可以只像读者一样看、圈、说感觉，就完成前三章校准；固定三问、必填拒绝原因和段落 Lock/Unlock 不再是默认体验。

## 7. S5 滚动自动连载与长篇治理

### 7.1 这阶段完成后长什么样

用户输入：

> 开始自动写吧。

仓颉：

> 好。我最多领先写 5 章。普通推进我会自己处理；涉及主角目标改变、重要人物死亡、世界规则变化或超过预算时，我会停下来问你。

右侧/任务页显示：

```text
正在写：第 18 章
已经完成：第 14–17 章
等待阅读：4 章
当前检查：人物有没有知道不该知道的事，前后时间是否对得上
本轮已花费：8.6 元
最近恢复位置：第 18 章场景 2
```

### 7.2 用户能做什么

- 一次准备 1–5 章；
- 对话查询进度、费用和暂停原因；
- 暂停、恢复和取消；
- 决定重大剧情变化；
- 退回较早章节；
- 保留旧分支；
- 查看受影响内容；
- 只重新生成受影响章节；
- 查看故事记忆中的人物、时间、规则和伏笔。

### 7.3 暂时不能做什么

- 不能无限领先；
- 不能预算越界后继续；
- 不能静默决定重大事件；
- 不能删除旧分支后无法恢复；
- 不能承诺 iPad 后台无限持续生成。

### 7.4 关键验收

1. 三章确认后启动持续创作；
2. 设置只领先两章；
3. 查询“现在写到哪了”，回答必须来自真实任务；
4. 说“先暂停”，任务在安全 checkpoint 停止；
5. 强退再打开后说“接着写”，不重复正文和费用；
6. 模拟重要人物死亡，仓颉必须暂停并解释后果；
7. 修改旧章，旧版本仍存在；
8. 影响分析能指出后续哪些章需要处理；
9. 只重生成受影响部分；
10. 连续至少五章无已知硬性人物知识、世界规则和时间线冲突。

### 7.5 阶段退出标准

用户能像指挥 Agent 一样用自然语言持续控制小说生产，后台保持版本、预算、恢复和长篇一致性。

## 8. S6 质量、增强资料、导出与候选正式版

### 8.1 这阶段完成后长什么样

用户可直接说：

- “把这个 PDF 当参考资料。”
- “查一下古代科举制度，别写错。”
- “检查有没有科技词污染修仙世界。”
- “把前三章导出成 DOCX。”
- “备份这本小说。”
- “清理技术日志。”

仓颉完成后用大白话报告，详细来源、失败页、版权边界和技术日志放入详情。

### 8.2 用户能做什么

- 在 S3 的 TXT/MD/文本 PDF/URL 基础能力上，导入 DOCX、扫描 PDF 和 ZIP；
- 查看每个文件成功、部分成功、需校对、不支持或失败；
- 使用离线 OCR，并查看置信度和失败页；
- 对 ZIP、大文件、异常文件名和同名碰撞执行安全清点；
- 搜索并保存研究卡；
- 运行中文网文专项审校；
- 导出 TXT/MD/DOCX；
- 导出加密项目包；
- 开启 Face ID；
- 清理日志；
- 调整字号、横竖屏和无障碍选项。

### 8.3 首版仍不包含

- 不自动登录番茄、起点等平台，也不自动发布；
- 不承诺 iPhone、Mac、多端同步或云端协作；
- 不承诺 iPadOS 后台无限连续生成，切后台时仍以安全保存和回来后恢复为准；
- 不允许绕过预算上限的无人值守写作；
