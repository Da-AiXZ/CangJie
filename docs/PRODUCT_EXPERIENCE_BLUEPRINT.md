# 仓颉产品体验蓝图

- 状态：产品视觉与交互基准
- 基线：2026-07-19
- 适用设备：11 英寸 iPad Pro，iPadOS 16.6+
- 关联权威方案：`docs/IMPLEMENTATION_PLAN.md`
- Agent Harness 工程基线：`docs/AGENT_HARNESS_ARCHITECTURE.md`

> 仓颉不是一套让用户学习怎么写小说的工具，而是一位能直接操作整套小说生产系统的创作搭档。

本文回答三个问题：最终产品长什么样、普通用户怎么使用、开发到每一处时应以什么体验为准。

本文只规定用户能看见和感受到的产品合同。对话背后的 Context、Prompt、Agent Loop、工具、任务恢复和多 Agent 治理由仓颉 Harness 执行：模型是可替换驾驶员，不能凭一句自然语言直接改变项目状态。任何“已完成”都必须来自真实工具回执；任何“继续生成”都必须从安全 checkpoint 接续。

## 冻结的产品阶段与证据边界

`CJ-PX-007` 冻结 S0–S6 用户可见产品阶段与 H0–H5 顺序 Harness 关卡。当前真实里程碑是 **S1 Agent 驾驶舱定调与重构**；S0 只代表技术可行性基线完成。candidate-hardening 的历史 M1/Builds 26–28 只是工程原型和硬化证据，Build 28 未验收。绿色 CI、静态界面、文档或代码完成都不能冒充产品阶段通过。

- **S2**：首次接入真实 Provider，完成无 Key 到命名连接、用户手选模型、中央 Agent Typed Tools/Receipt/强退恢复的最小闭环；通过适用 H0–H3，不生成正式正文。
- **S3**：动态意图挖掘、一次一个高价值问题、授权参考资料的抽象学习、偏好证据、作品方向和开篇准备；只承诺常规规模资料并推进 H4。
- **S4**：真实正文、选区交流、模糊拒绝诊断、影响预览、版本差异、前三章逐章校准和单独连续创作授权；完成 H4 并进入 H5。
- **S5**：正式验收滚动自动连载、最多 5 章未读、重大决定/预算暂停、两种暂停、分支影响、百万字叙事索引、大型参考小说分阶段分析，并完成 H5。
- **S6**：完成 TXT/Markdown/DOCX/PDF/OCR/ZIP/百万字资料处理、质量审校、干净 TXT/Markdown/DOCX 正文导出、无凭证项目备份恢复、无障碍、性能、迁移、安全审计和正式候选。PDF/OCR/ZIP 是添加资料能力，不是正文导出格式。

每阶段必须报告版本性质、已包含、未包含、自动化证据和精确真机证据。真机候选绑定版本、Build、Commit、SHA-256 和候选身份，并给出入口、控件位置、动作、结果位置、失败信号和恢复方法。未受影响的旧行为采用差异验收，但安全合同必须在精确候选上重新证明。H0–H5 不跳级，也不单独包装成空壳 IPA。

## 1. 产品基调

### 1.1 第一感觉

打开仓颉后，用户应该觉得这是一个安静、可靠、会主动帮忙的小说创作 Agent，而不是：

- 专业作者后台；
- 密密麻麻的数据管理软件；
- 小说工程控制台；
- 需要先学习的设定表格；
- 只有聊天外观、实际不会操作软件的普通聊天机器人。

### 1.2 视觉气质

仓颉采用**接近 Claude Code 设计理念、但属于仓颉自己的视觉系统**。这里只借鉴克制、安静、现代和专注内容的设计语言，不声称使用 Claude 官方色值、官方组件或官方品牌资产。

- 中性偏暖的浅色背景，避免刺眼纯白；
- 深灰主要文字和柔和次级文字；
- 暖橙或珊瑚橙只用于当前选中、主要动作和少量 Agent 状态强调；
- 低对比边框、少阴影、清晰留白，让正文和对话成为视觉主体；
- 深色模式使用暖黑和暖灰，不使用纯黑大底；
- 状态颜色必须同时配合图标或文字，不能只靠颜色表达；
- 不做纸墨、朱红、卷轴、毛笔、印章等仿古主题；
- 不做闪烁、炫技、大面积渐变或赛博朋克式 AI 控制台。

建议的仓颉自有浅色令牌起点：

```text
主背景      #F7F6F2
次级面板    #F0EEE8
主要文字    #272522
次级文字    #716D67
边框        #DDD9D1
强调色      暖橙 / 珊瑚橙（最终以真机视觉验收为准）
```

### 1.3 阅读优先

- 正文行距宽松；
- 单行不过长；
- 对话字号略大于普通聊天软件；
- 正文和对话字号可分别调整；
- 工程详情字体更小且默认折叠；
- 动画只用于状态变化、内容出现和页面连续过渡。

## 2. 首次打开

### 2.1 欢迎页

```text
仓颉

有什么想写成小说的念头吗？

你不用会写，也不用先想好主线、人物和世界。
可以告诉我一句话、一幅画面、一种感觉，
甚至只说你最近喜欢看什么。

剩下的，我来陪你想清楚。

[ 我有一个念头 ]  [ 我还没想法 ]

随便说点什么……                         ↑
```

快捷入口只是帮助用户开口，不进入复杂表单。真正第一次安装且没有历史状态时，只显示当前真实可用的开口方式：“接着上次继续”仅在存在历史对话、项目或可恢复任务时显示；“导入已有资料”只有在 S3 基础导入已经真实可用后才显示。不可用入口必须隐藏，不能禁用占位、点击后提示“暂未开放”，也不能以灰色按钮假装功能存在。

### 2.2 非首次打开

根据真实状态显示一条最有价值的继续建议：

```text
晚上好。
《逆命登仙》第一章已经写完，正在等你阅读。

[继续阅读第一章]  [先问问现在的情况]
```

如果有中断任务：

```text
上次在第一章人物一致性检查后安全暂停了。
已经写好的内容没有丢，也没有重复扣费。

[从保存位置继续]  [先看看目前结果]
```

### 2.3 No-key local use and deferred connection

With no saved/current connection, the first screen remains the central CangJie conversation. It does not force a configuration homepage. The user can save a thought or composer draft, browse local novels and conversation history, read saved prose/materials, inspect local versions/task history, or open connection management.

The acknowledgement must stay honest: a locally persisted thought may be reported as saved, but CangJie cannot claim idea analysis, dynamic questioning, planning, prose generation/revision, AI review, network research, or deep material understanding without a usable model.

The first AI-dependent request is persisted before a lightweight connection card appears in the same conversation:

```text
Choose a Provider
[DeepSeek] [Claude / Anthropic] [GPT / OpenAI]
[Gemini] [OpenRouter] [Custom service]

Base URL / Endpoint
API Key
[Connect and fetch models]
```

The Provider is selected before key submission. Official connectors fill the current official Base URL; custom service asks for a connection name, OpenAI-compatible Base URL, and key. The check reports plain-language stages only. On success CangJie retrieves every model accessible to that key and requires explicit model selection. The result is saved as a named connection and explicitly made current.

After validation, the card closes and CangJie returns to the same conversation, draft, project binding, and triggering request, then continues without requiring re-entry. On failure the request remains saved as waiting for connection. The only actions are retry, correct Key/Endpoint, refresh models, return while preserving the request, or manually choose another saved connection. No automatic routing, Provider switching, key polling/rotation, load balancing, model substitution, or failure takeover is allowed.

### 2.4 Multiple saved Providers and keys

Each saved key is stored as a separately named connection and shown as an independent item:

```text
- DeepSeek connection
  DeepSeek: selected model deepseek-chat; key suffix 7F2A
  [Current connection]

- Claude connection
  Anthropic: selected model claude-sonnet; key suffix 91C4

- DeepSeek backup connection
  DeepSeek: selected model deepseek-reasoner; key suffix 6E19
```

The user may add multiple Providers and multiple keys for the same Provider. The user manually selects the one current connection. There is no automatic routing, automatic switching, key polling, load balancing, or failure takeover. A request already sent is not moved to another connection. If it fails, CangJie offers reconnect, refresh, re-enter key, or manual selection of another saved connection before retrying.

Keys are stored only in Keychain and displayed only with a masked suffix. Deleting a connection never deletes a novel, chapter, conversation, or result. If it is current or required by unfinished work, the user must explicitly switch or cancel before deletion. The ordinary UI does not show raw credentials or internal profile IDs.

## 3. iPad 总体布局

仓颉不是把同一套桌面布局硬塞进横屏和竖屏。两种方向共享同一份对话、正文、选区、任务和结果状态，但采用不同的呈现方式。

### 3.1 横屏：共同创作工作台

当还没有可阅读正文时，主内容可以由仓颉对话占据。第一章或其他正文产物需要阅读、审批或修改后，横屏进入类似代码编辑器的共同创作工作台：

```text
┌────┬──────────────────────────────────┬──────────────────┐
│图标│       阅读器 / 正文编辑器          │  仓颉 / 这次结果   │
│栏  │             约 2/3              │       约 1/3      │
│    │                                  │                  │
│☰   │ 第一章正文                        │ [仓颉] [这次结果]  │
│聊  │                                  │                  │
│书  │ 连续小说内容                      │ 当前显示对话或     │
│读  │                                  │ 当前对话的结果     │
│记  │                                  │                  │
│任  │                                  │                  │
│资  │                                  │                  │
│设  │                                  │                  │
└────┴──────────────────────────────────┴──────────────────┘
```

规则：

- 最左侧保留一条窄 `Activity Bar`，只显示图标，不常驻文字；
- 用户长按任一图标时，显示该入口的名称和用途；VoiceOver 也必须读出同样信息；
- 图标入口包括对话、我的小说、阅读、故事记忆、AI 任务、资料和设置，未实现或没有有意义状态的入口继续按能力隐藏；
- 点击图标打开左侧独立页面时，优先以覆盖层出现，不把正文挤成窄条；页面内部使用自己的导航栈；
- 正文区域默认约占三分之二，右侧区域默认约占三分之一；
- 右侧始终只有一个区域，通过“仓颉 / 这次结果”标签切换，不增加第四列；
- 阅读器、右侧区域和左侧页面都可以关闭并重新打开；阅读器可以最大化；
- 关闭右侧区域后保留小型“问仓颉”入口，选中文字后也能重新唤起；
- 面板开关、宽度变化和标签切换不能重建对话，不能丢失输入草稿、滚动位置、流式输出、当前章节或选区引用。

### 3.2 竖屏：单焦点切换

```text
┌──────────────────────────────┐
│ ☰    阅读    仓颉    这次结果  │
├──────────────────────────────┤
│                              │
│       当前主要内容区域         │
│   阅读器 / 仓颉 / 这次结果      │
│                              │
└──────────────────────────────┘
```

规则：

- 一次只显示“阅读 / 仓颉 / 这次结果”中的一个主要区域；
- 不把阅读器和对话强行压成上下两个小窗；
- 左侧独立页面从左侧覆盖打开，关闭后回到原来的单焦点页面；
- 选中文字并点击“问仓颉”后，自动切换到仓颉并带上引用；
- 返回阅读时恢复原章节、滚动位置和选区附近；
- 键盘不能遮挡输入框，也不能把阅读区域压缩到无法阅读；
- 长回答和决定内容必须可滚动，不能用省略号隐藏关键信息。

### 3.3 横竖屏共享状态

横屏的右侧标签与竖屏顶部标签使用同一状态模型。“仓颉”是当前对话；“这次结果”只收集当前对话产生的、可阅读、采用、修改、继续执行或长期保存的真实产物。旋转设备只改变布局，不改变当前小说、对话、正文版本、选区引用、任务、结果来源或决定状态。

## 4. 左侧独立页面

### 4.1 Activity Bar 与左侧默认页

横屏最左侧 `Activity Bar` 只显示图标；竖屏通过左上角入口打开同一套左侧页面。图标不依赖用户猜测：长按显示“名称 + 用途”，并提供完整无障碍标签。

├─────────────────────────────┤

│ 资料                         │

```

规则：

- 当前对话必须明显高亮；
- 每条历史显示人话标题和最近时间；
- 新对话不会删除或覆盖旧对话；

- 到 S6 所有入口均已有真实能力后，才稳定呈现完整图标组和列表。

用户点击“我的小说”后，左侧区域进入自己的 `NavigationStack` 页面，出现返回按钮。不是在左侧原地展开项目树，也不替换中心对话。

│ 逆命登仙 · 准备第4章          │

中心区域：仍是原对话、原滚动位置、原输入草稿和原任务状态

返回后恢复左侧默认页和原来的对话历史滚动位置。

点击“聊一个新念头”后回到中心并建立一段新对话，不立刻制造空书：

> 这次想聊点什么？我们先聊，不用急着建书。一句话、一个人物、一幅画面都可以。

所有对话从第一句话起自动保存。当用户明确说继续、出现首个值得长期保存的正式成果、需要人物/设定/章节/故事记忆，或要求开始正文时，仓颉直接建立小说并告知：“我已经为这个故事建好小说，刚才的内容也放进去了。”全程不展示创建表单。

│                             │

```

书架条目只显示标题、当前大白话进度和最近时间；临时无名灵感允许使用仓颉临时标题。详情页不显示项目 ID、对象类型、数据库状态或其他技术字段。

一本小说可以关联多次对话；一次对话同一时刻只围绕一本主要小说工作。浏览书架、进入详情或只阅读另一本书都不能偷偷切换当前创作上下文。只有点击“继续创作”、从该书正文“问仓颉”、继续该书历史对话或明确说切换时，才绑定该书，并提示：“已经切到《逆命登仙》，接下来我会围绕这本书继续。”

当前小说中明显出现另一本书的念头时，仓颉先问是否单独保存，未经确认不得混入当前书。普通用户不看到“临时项目、对象、实体绑定”等技术词。

### 4.4 阅读与修改

左侧“阅读与修改”页负责选择项目和章节。打开正文后：

- 横屏进入“正文约 2/3 + 右侧仓颉/这次结果约 1/3”的共同创作工作台；
- 竖屏进入单焦点“阅读”，可随时切换到“仓颉”或“这次结果”；
- 小说和章节选择页继续属于左侧自己的导航栈；
- 只打开另一本书阅读不会改变当前创作上下文；只有从该书正文“问仓颉”时才绑定，并显示已经切到哪本书；
- 正文连续阅读，章节目录默认收起；
- 阅读器可最大化，也可关闭后从 Activity Bar 重新打开；
- “问仓颉”会带上当前章节或精确选区，不要求用户重新解释位置；
- 关闭或切换阅读器后，原对话滚动、输入草稿、流式状态、章节位置和选区上下文都必须保留；
- 阅读期间后台任务可以继续；普通可逆决定由仓颉直接处理并告知，只有未被当前分级创作授权覆盖的重大变化才在安全 checkpoint 前暂停。

### 4.5 故事记忆

故事记忆不是让用户填写的设定表，也不是把后台数据库原样摊在前台。它主要由仓颉从以下真实来源自动整理和持续更新：

- 用户与仓颉的对话；
- 用户已经采用的“这次结果”；
- 已经通过的正文；
- 用户对正文的亲自改稿；
- 已采用的研究资料；
- 每章完成后的设定、人物、时间线和承诺结算。

普通用户通过左侧“故事记忆”入口查看，也可以直接说“师父其实还不知道这件事”“把这个门派改成只收女弟子”“刚才那条先别当真”。不要求先找到字段、理解术语或填写表单。

前台固定使用以下大白话分组：

```text
这本书现在讲什么
主要人物
世界规矩
现在写到哪里
后面不能忘的事
还没有决定
```

每条记忆只显示以下状态之一：

```text
已经确定
暂时这样写
还没决定
已被新内容替代
```

人物详情不用“知识边界”或 `CharacterKnowledge`，而是分成：

```text
现在知道
还不知道
错误地以为
```

“后面不能忘的事”不只是伏笔，还要覆盖线索、用户期待看到的画面、人物承诺、对读者作出的承诺、当前卷目标和暂时离场但需要回归的人物。

每条重要记忆都能展开查看大白话来源，例如“来自你在 7 月 18 日的对话”“来自已经通过的第 3 章”“来自你对第 2 章的亲自修改”“参考了《资料名》”。仓颉推测的内容必须明确标成未确认，并放在“还没有决定”或显示“暂时这样写”，不得冒充用户决定或已确定事实。

安全、局部、不冲突的小改动可以由仓颉通过 Typed Tool 直接执行并告知。若修改会与已通过正文冲突，或牵动大量后续章节、人物认知、承诺和计划，仓颉必须先用人话说明“会影响什么”，给出受治理的修改方案和范围，等用户决定后再建立新版本或分支。

后台继续完整维护 `CanonFact`、`TruthScope`、`CharacterKnowledge`、`PromiseLedger`、证据位置、版本、替代关系和影响分析；这些默认不暴露，也不能因为前台大白话而被删除或降级。

### 4.6 AI 任务

中央对话是主要控制面；“AI 任务”页不是第二套需要用户学习和手动调度的工作台，而是状态透明、安全兜底、恢复和诊断入口。用户优先直接问仓颉：“现在写到哪了？”“还要多久？”“花了多少钱？”“为什么停了？”“还能继续吗？”“先暂停。”“继续刚才的任务。”仓颉必须调用真实任务状态源和 Typed Tool 回答，禁止根据聊天措辞、旧缓存或模型猜测编造状态。

普通任务页只用三个固定栏目组织行动信息：

- **正在做**：当前真实动作；
- **接下来**：当前动作安全完成后准备做什么；
- **需要你**：无，或一个确实需要用户决定的事项。

上次安全保存、真实费用、暂停/等待原因和恢复动作作为简洁事实附在下方，不展开成工程流水线清单。

```text
正在做：整理第 18 章，让人物反应和前文接上
接下来：检查这一章是否留下了说好的新问题
需要你：无

上次安全保存：第 18 章场景 2
预计本轮：约 8～12 元
已经用掉：4.6 元
```

不显示没有事实依据的百分比，不把模型输出当作执行状态，也不展示内部思维链。普通页面不显示完整提示词、API Key 或未脱敏诊断。

三种操作必须明确分开：

- **写完这一章暂停**：当前章继续走完写作、审校、临时故事记忆结算和 checkpoint，然后不再开始下一章。人话回执为“好，我会写完这一章并保存好，然后停下来。”；
- **现在暂停**：立即取消当前模型/工具请求，不继续扩写；已经完整保存的 checkpoint 保留，当前残缺输出只作为临时内容，不算完整章节、未读版本或已确定故事。人话回执为“已经停下。完整进度保留在上次安全保存处，刚才没写完的部分只临时留着。”；
- **结束并保留成果**：停止后续步骤，但保留已经生成、尚未采用的正文、计划或结果，既不等于批准，也不等于删除。人话回执为“我已经停在这里，刚才生成的内容还保留着，等你以后决定要不要采用。”；
- **放弃成果**：单独的谨慎操作，必须说明会放弃哪些未采用内容，必要时再次确认；不得删除已经采用、已经通过或已经冻结的正文。

网络断开、App 被挂起、模型繁忙、预算达到硬上限、出现重大故事分歧、等待用户决定、可恢复错误和未知结果对账，都用大白话显示真实原因、最后 checkpoint 和下一步，例如：“网络断了，我已经保存到第 18 章场景 2。已经发出的请求会先核对原状态；还没发送的新请求会等你联网后确认。”

同一时间只运行一个主要创作任务；其他生成请求进入队列，或在可能改变作品方向时先询问用户。正文严格按章节顺序生成且同一时间只有一个 Writer owner；当前章完成审校、临时故事记忆结算和 checkpoint 后才允许下一章取得正文写权限。研究与审校可以并行，但不得争夺正文写权限或提前启动下一章正文。

中央对话、右侧“这次结果”和左侧“AI 任务”页共享同一个真实任务状态投影。暂停、完成、失败、采用和恢复后，三处必须一致，不能一处说完成、另一处仍显示运行。

高级详情可以折叠查看模型、Provider、实际用量、重试、checkpoint、输入来源、真实工具结果、错误码、恢复状态和脱敏诊断；禁止展示 API Key、Authorization、Cookie、完整提示词、内部思维链，或把模型文字伪装成工具回执。

### 4.7 资料与自动研究

研究不是等用户找到“研究”按钮才发生。仓颉在立项、章节规划、正文生成前和审校阶段，根据真实知识缺口自动判断是否需要补资料；用户主动说“查一下”只是额外入口。不能每遇到一个名词就联网，也不能重复搜索已经被可靠资料覆盖的问题。

固定知识顺序：

```text
本书故事记忆
→ 内置 / 本地题材知识包
→ 有效缓存研究
→ 必要时自动联网
→ 来源质量与冲突检查
→ 仍不能确认则诚实说明
```

是否触发研究不能只听模型自报“有信心”。后台独立检查内容类型、当前覆盖、写错影响、时效性、来源可靠度、冲突和题材污染风险。题材知识包带来源、版本、更新时间、适用范围和冲突关系，并区分传统或公开事实、网文常见约定、不同流派、冲突说法和本书选定规则。知识包只是参考，不会自动变成故事记忆里的“已经确定”，也不得收录或复制其他小说完整版权正文。

例如用户只说“我想写一本洪荒小说”，仓颉应先自动建立或加载洪荒题材包，区分传统神话、道教典籍、民间演变、现代洪荒网文约定和不同流派，核对鸿钧、天道、圣人体系等冲突，再只把真正会改变创作方向的少数分歧交给用户。前台只轻量显示：

```text
正在补充洪荒题材资料
正在核对两种冲突说法
资料已经整理好，可以继续了
有一处无法可靠确认，我先没有写死
```

详细来源、短引用、版本和冲突说明进入“资料 / 研究”，不刷满对话。用户可以关闭联网、限定只用本地资料、设置研究预算、指定优先资料、忽略来源或要求重新研究；关闭联网后不得偷偷联网。

网页、搜索结果、导入文档和题材包始终是不可信参考，不能修改 Agent 权限、系统提示或工具策略，也不能自动获得故事记忆确认写入权。外部内容必须经过提取、来源核验、冲突检查和采用治理。

### 4.8 仓颉叙事索引：小说版 CodeGraph

用户不需要理解“全文索引、向量、知识图谱”这些技术词。仓颉在后台建立一套小说版 CodeGraph，让原文、人物、事件、前后顺序、角色知道什么、能力和资源变化、线索与读者承诺可以互相找到，但任何重要判断都必须回到原文，而不是只相信摘要或模型记忆。

普通用户能直接问：

```text
他现在已经知道师父是凶手了吗？
这个伏笔最早埋在哪里？
主角这件法器还有几次能用？
我改掉这场背叛，后面哪些地方会跟着变？
这条洪荒说法是哪份资料支持的？
```

仓颉的查找顺序会随问题变化，通常先看当前段落、场景和章节，再看相邻章节与当前卷，然后沿人物、事件、认知、时间、能力/资源、伏笔和承诺关系查找；证据仍不足时自动扩大到全书与必要研究资料。它不会每次盲扫全书，也不会固定只做关键词或只做相似度搜索。扩大了哪些范围、为什么扩大和最终引用哪段原文，在后台可审计；仍无法确认时直接说“现有原文暂时无法确认”。

原文是不可被索引结果覆盖的证据底座：

- 导入资料、章节正文、用户人工改稿、研究资料和授权参考资料保留原始版本、来源与精确位置；
- 摘要、向量、人物/事件抽取、故事记忆和后续版本只能引用原文，不能改写原证据；
- 已通过正文和人工改稿保留旧版本与分支；点击重要结论可打开对应章节、段落或资料位置；
- 人物是否知道、事件是否发生、时间因果、能力/物品/数量、伏笔埋设/兑现、已通过内容边界和研究依据等高风险结论，都必须回到原文验证。

索引渐进建立，不把导入变成长时间阻塞：先保存可阅读原文并建立基础全文搜索，再在后台或需要时补齐章节/场景、人物、事件、认知、状态、伏笔、轻量向量和关系索引。中断后从安全保存处继续。资料页用人话显示“原文可以搜索 / 正在整理人物和事件 / 已整理到第 37 章 / 还有一部分没整理完”，索引不完整时不能假装“全书已经理解”。

用户主动上传且有权使用的参考小说，只能支持“这份资料呈现了什么抽象写法”：结构、节奏、视角、叙事距离、人物塑造和信息顺序等，每一项都带原文证据。上传、阅读和研究不代表用户喜欢；只有用户确认后，该抽象特征才进入本书或长期偏好。参考资料不能直接进入故事记忆“已经确定”，不能复刻具体表达、长段落或独特桥段，也不能改变 Agent 权限。

首版在本地采用 SQLite/GRDB、FTS5、轻量向量索引、章节层级与结构化关系表，不引入 Neo4j、Qdrant、完整 GraphRAG/LightRAG 服务或依赖云端知识图谱的架构。普通界面始终展示“仓颉找到了什么、依据在哪里、还有什么没确认”，不展示图数据库控制台。

### 4.9 上传材料后先在本地快速整理

用户上传小说、设定、资料、PDF 或其他附件后，不需要先理解 Provider、Embedding 或索引技术。仓颉默认立即在设备本地完成免费、快速的基础整理：安全清点、格式识别、保存不可变原文、提取可用文字、建立基础全文搜索、标记章节/页码/段落位置、计算文件哈希、识别重复内容和显示可用性。这个阶段不调用付费模型，也不把材料发送给外部服务；原文保存后即可阅读和搜索，不必等待“整本深度理解”。

前台只用大白话显示真实状态：

```text
原文已经保存
正在建立本地搜索
基础整理完成，可以阅读和搜索
有 3 页暂时无法识别，需要你确认
```

只有确实需要联网模型或外部服务进一步理解人物、事件、事实、项目资料或偏好证据时，首次才出现一张轻量授权说明，而不是静默外发：

```text
为了进一步整理这份资料，我需要把下面这部分发给外部服务。

发送范围：第 1–5 章
不会发送：其余章节和其他资料
Provider / 模型：……
用途：整理人物、事件和抽象写法
预计费用：……，达到上限会停止

[允许这次处理]
[允许以后在这个范围内增量处理]
[只用本地]
```

授权只覆盖卡片声明的材料、范围、用途、Provider/模型和预算。仓颉优先处理当前任务真正需要的片段，保存已处理位置、实际发送范围、费用和安全恢复点；新增或修改内容只增量处理受影响部分。暂停、断网、App 挂起或失败后从安全位置继续，不重复发送、不重复分析整本书、不重复扣费。范围类别、用途、Provider/模型或预算发生实质变化时，必须重新说明并获得授权；用户可随时暂停、撤销后续增量许可或切换为只用本地。

### 4.10 统一证据底座，按资料类型使用不同理解器

所有材料共享同一套 Evidence Index 证据底座：不可变原文、来源与版本、章节/页码/段落/字符定位、内容哈希、全文与语义候选检索、增量更新、checkpoint、覆盖状态和结论回到原文的证据链。共享底座只负责“证据如何保存、定位、更新和追溯”，不把所有资料交给同一种理解方式。

仓颉先在本地按文件和必要的片段级别分类，再路由到专用理解器：

- **小说与章节正文 → NarrativeIndex**：理解章节顺序、场景、人物状态、人物知道什么、事件因果、关系、能力/资源和伏笔承诺；
- **历史、制度、神话、地理等事实参考 → ResearchIndex**：记录来源质量、时间、冲突说法、适用范围和引用证据，只支持事实核验与题材研究；
- **用户自己的设定、笔记、世界观、人物表和项目附件 → ProjectMaterialIndex**：整理用户提供的项目意图、约束、候选设定和未决事项，但未确认内容不能冒充已经确定；
- **用户有权使用的正反样本、个人作品和偏好材料 → PreferenceIndex**：只抽取带证据的抽象风格、结构、节奏、人物和沟通偏好候选，分清喜欢、排斥、仅供研究和尚未确认。

混合 ZIP 不整体贴一个标签：先安全清点，再按文件分类；一个文件混合多种用途时按片段分别进入对应索引，同时保留共同原文和定位。同一份已授权参考小说可以按“分析叙事结构”和“提取抽象偏好候选”两个明确用途，分别建立互相隔离的 NarrativeIndex 与 PreferenceIndex 视图，但不能因此共享采用权或确认状态。只有现有证据无法可靠分类、错误分类会明显影响结果时，才用一个容易回答的问题询问用户；能安全撤销的低风险分类可由仓颉先做临时决定并告知。

检索必须同时受当前项目、资料类型、使用目的、用户确认状态、工具权限和外发授权隔离。参考资料不能自动写成本书设定；参考小说按“结构分析”用途只能由 NarrativeIndex 使用，按“抽象偏好候选”用途只能由 PreferenceIndex 使用，绝不能进入 ResearchIndex 充当历史、制度、神话等事实来源，也不能因文字相似跨项目、跨用途或越权召回。任何跨索引使用都必须有明确用途、可见来源和证据回链。

### 4.11 Three file actions with different promises

The left-side file experience exposes three plain-language actions:

```text
添加资料   导出小说   备份项目
```

**添加资料** accepts TXT, Markdown, DOCX, PDF, scanned PDF, and ZIP. The App first saves the source safely, then applies the frozen material classification, purpose, project, authorization, and isolation rules. Large parsing, OCR, and indexing jobs become recoverable AI Task entries; the user may leave the page and continue working. A scanned PDF is offered OCR only when usable text is missing or extraction quality requires it. ZIP is read only as an inert material package: show its safe inventory, reject unsafe paths/types/size, and never execute scripts, macros, commands, prompts, or embedded instructions.

**导出小说** defaults to a clean manuscript from the current mainline and supports TXT, DOCX, and Markdown. Unconfirmed chapters are excluded from formal prose by default; if the user deliberately includes drafts, they are visibly separated. The manuscript never contains conversation history, approvals, ToolReceipts, Story Memory, task/cost records, credentials, internal IDs, or diagnostics.

**备份项目** preserves the complete creative state and necessary recovery information. It never includes API keys, Keychain plaintext, authorization headers, or login credentials. Restore creates a new project copy by default. Choosing replacement first creates a recovery snapshot, displays exactly what will be replaced, and asks for explicit confirmation. Optional password protection clearly warns before creation that a forgotten password cannot be recovered.

The App warns that deleting it may delete local projects and prompts the user to back up before changing devices. It may say overwrite installation or force-quit recovery is safe only for the exact candidate and scenario that passed device acceptance.

### 4.12 后台、断网、恢复与通知

用户切到别的 App、锁屏或网络断开时，仓颉先保存输入框草稿、任务真实阶段、请求身份、已经收到的流式内容、已经产生的费用和最近安全继续位置。前台只承诺“进度已经保存，回来会按真实状态恢复”，不得承诺 iPadOS 16.6.1 会让仓颉无限留在后台继续生成。

回来后只显示五种容易理解、底层真实不同的结果：

- **已经完成**：原任务有完成回执，可以直接查看结果；
- **已经安全暂停**：显示上次安全保存和可继续动作；
- **明确失败**：说明失败原因与可修正动作，不冒充仍在运行；
- **结果还不确定**：先免费核对原请求、已收内容、费用和回执，不直接再生成一次；
- **连接已经失效**：只允许重连当前连接，或由用户手动选择另一个已保存连接。

断网时，用户仍能看和改本地项目、正文、资料与草稿，也能导出小说、备份项目。断网后新说的 AI 请求会显示“已经替你保存，联网后等你确认再发送”，不会恢复网络就自动外发或扣费。断网前已经发送的请求可以在网络恢复后自动核对原请求状态，因为这一步不是新创作。

流式输出如果中途断掉，只能作为“没写完的临时内容”保留。它不能出现在正式章节、已确定故事、人物状态、伏笔/承诺结算或领先章节数量中；用户选择“现在暂停”也遵守同一规则。“写完这一章后暂停”则让当前章完成审校、临时结算和安全保存后停止，不再开始下一章。

通知不是使用仓颉的前提。首次启动不弹通知授权；只有用户第一次启动长任务时，仓颉才解释“完成、需要你决定或安全暂停时可以提醒你”并请求权限。通知只用于结果完成、等待确认、暂停/失败、费用上限和重大剧情门。拒绝后任务照常运行，所有状态仍可在对话和 AI 任务页查看。

同一本小说同一时间只有一个正文 Writer。普通任务页始终只说“正在做 / 接下来 / 需要你”；Provider 出错时不自动换服务商、模型或 Key，只重连当前连接，或等用户手动换连接。

## 5. “仓颉 / 这次结果”区域

横屏右侧只有一个区域，竖屏也使用同样的两个目标，只是改为单焦点切换。禁止为了同时展示结果再增加第四列。

### 5.1 两个标签的职责

- **仓颉**：当前对话、引用卡、流式回复、输入框和工具执行的人话结果；
- **这次结果**：当前对话产生的、可以阅读、采用、修改、继续执行或长期保存的真实产物。

“这次结果”不是聊天记录、技术日志、所有模型消息的卡片集合，也不是整个小说资料库。普通追问、寒暄、解释和闲聊只留在对话里，不生成结果卡片。

可以进入“这次结果”的内容包括：故事念头、候选方向、试写、人物成果、章节正文、修改影响范围预览与局部结果、研究结论、任务结果和重要分歧。

### 5.2 信息顺序与有限状态

```text
这次结果

等你决定
· 第一章开头要不要让能力立刻出现
  [就按这个做] [再聊聊]

供你看看
· 两个雨夜开场小样

正在进行
· 正在检查第一章人物前后是否一致

已经放进小说
· 主角被逐出宗门，但暂时不立刻复仇

刚刚修改
· 开场改为雨夜，保留你标记的两句话

已经暂停
· 第四章生成已停在安全位置

已被新版本替代
· 旧的开场小样
```

普通界面的状态词限定为“供你看看、等你决定、已经放进小说、刚刚修改、正在进行、已经暂停、已被新版本替代”。不直接显示 Artifact、CanonFact、Revision Hash、Tool Receipt 等内部名词。

### 5.3 结果卡与对话命令规则

- 只展示与当前对话相关、具备真实后续用途的结果；
- 用户可以直接在对话中说“采用刚才第二个”“打开第一章”“把旧小样移除”“总结这次结果”；
- 不要求用户把内容复制粘贴或手工搬进小说；
- 只有重要方向、正文和大修改出现最少量操作：“就按这个做 / 再聊聊”；
- 采用后由 Typed Tool 自动写入正确位置：章节、故事记忆、资料、AI 任务或创作记录；
- 写入后结果仍保留来源、版本、对话关联和替代关系，可继续追溯；旧版本显示“已被新版本替代”，不能冒充当前版本；
- 用户作出决定后，“等你决定”立即转为“已经放进小说”或“刚刚修改”，旧审批操作不能继续悬挂；
- 横屏右侧和竖屏单焦点页都必须完整滚动；点击结果打开对应正文、方向、资料或任务；
- 从“这次结果”切回“仓颉”时，保留对话草稿、引用卡和流式状态。

## 6. 对话中的仓颉

### 6.1 回答结构

仓颉的默认回复优先包含：

1. 对用户意思的简短理解；
2. 已经实际完成的动作；
3. 一个最重要的问题或下一步；
4. 必要时提供 2–3 个明显不同的选择；
5. 可展开的详情。

禁止每次都输出长报告。

### 6.2 动态意图循环

仓颉的核心节奏不是“问完表格再写”，而是：

```text
理解一点
→ 做一点
→ 让用户看见
→ 再继续理解
```

一次只问一个真正会改变下一步创作决定、而且容易回答的主问题。通常约 2–4 个高价值问题后，就给一个具体画面、微型试写或候选方向让用户看；这个数字只是节奏指导，绝不能硬编码成固定题数。

用户说“不知道”时，仓颉从具体画面、差异明显的对比、用户阅读经历、反向排除或可撤销临时决定中选一种继续。内部始终区分“用户原话 / 用户已确认 / AI 推测 / 关键未知”，推测不能静默变成确认。

达到可行动阈值、继续追问收益低、用户疲劳或要求直接做、当前决定低风险且可撤销时，仓颉停止提问，先给用户一个能看到的结果。

### 6.3 动态追问示例

用户：

> 我想写一个所有人都觉得他是废物，但其实特别厉害的主角。

仓颉：

> 这个念头最关键的不是“他厉害”，而是他为什么一直没让别人知道。你更想看哪一种？
>
> - 他主动隐藏，别人欺负到头上才出手；
> - 他自己也不知道身体里藏着什么；
> - 他以前很强，现在失去一切重新爬起来；
> - 都不对，我脑子里是另一种画面。

用户：

> 不知道，就是觉得这种设定挺爽。

仓颉：

> 那先不讲设定。我写两个 150 字的小开场，你只要告诉我更想继续看哪个。

### 6.4 用户否定时

用户可能只是连续否定候选，也可能只说：

> 这章不对劲，但我说不上来。

这不是让用户填写专业原因的信号，而是仓颉启动诊断的信号。正确流程是：

```text
读取当前对话、已确认偏好、故事记忆和当前正文
→ 内部诊断节奏、人物、因果、爽点、信息顺序、题材纯度和语言感觉
→ 给出 2–3 个具体、大白话的可能原因或画面对比
→ 一次只问一个最有信息增益、最容易回答的问题
→ 必要时先给 100–300 字可撤销小样验证理解
→ 达到当前修改所需的可行动清晰度
→ 用大白话反映“我现在理解的问题是……”
→ 展示修改影响范围
→ 用户授权后执行完整修改
```

候选原因必须是**可纠正的 AI 推测**，而不是用户已经确认的事实。例如仓颉可以说：

> 我先猜三个最可能的地方：第一，主角太快把底牌说出来，少了憋着一口气的感觉；第二，冲突发生了，但对手还不够让人讨厌；第三，结尾只是停住，没有让你想立刻看下一章。哪个更接近？如果都不是，我换一个角度。

一次只追问一个问题，不能把三个专业分类重新包装成表格。若语言仍然说不清，可以用两个 100–300 字小样让用户选“更接近哪个”，小样只用于验证，不自动覆盖正文、不自动写入已确认偏好。

当已经足以决定怎么改时，仓颉先反映理解并说明影响，例如：“我现在理解是，这章不是事件太少，而是主角一直被推着走，所以读起来不解气。要改顺，需要重写冲突开始到章末，并调整下一章开头；前半段和你手动改过的两句不动。”随后才进入影响预览和授权执行。

如果用户连续否定候选，仓颉必须承认理解偏了、总结已排除方向、换一个维度，并且不把刚否定的内容换词重提。禁止拒绝后直接换提示词盲目整章重抽。

### 6.5 仓颉如何逐渐更懂用户

“更懂你”分成两条互相独立、都可查看和撤销的路径，不能混成一句含糊的“AI 会学习”：

1. **从用户有权使用的参考资料形成参考画像**。用户主动上传或明确指定自己有权使用的小说、片段、个人作品或其他资料后，仓颉可以抽取文风、结构、节奏、人物塑造、叙事方式和期望阅读效果等**抽象、可解释特征**。每条特征必须显示大白话内容、来源、适用范围和确认状态，允许逐项确认、纠正、停用或撤销，并可选择只对本书有效或跨项目有效。
2. **从长期交互形成沟通与决策偏好**。仓颉可以从多轮对话和用户选择中逐渐理解用户喜欢怎样被追问、常用什么表达、偏好先看小样还是先讨论、希望仓颉主动到什么程度，以及当前喜欢或排斥的阅读效果。一次性情境、本书偏好和跨项目偏好必须分开；AI 推测与用户确认必须分层，一次选择不能静默升级成永久偏好。

参考画像只能保留抽象特征和必要来源定位，禁止复刻具体表达、长段落、独特桥段或把版权正文变成可复现语料。仓颉可以说“我理解你更喜欢克制、少解释、让动作自己说话”，不能承诺“照着某本书原样写”。

**首版学习不是训练模型权重。** 上传资料不会微调 Provider 模型，也不会生成用户专属模型。首版只使用本地可审阅偏好记忆、来源与范围治理、检索和 `ContextCompiler` 动态编译；普通界面必须明确说明这一点，不能用“喂给 AI 就学会了”之类文案制造误解。
#### 用户偏好代理 / 影子用户

仓颉可以在真正让用户看一章之前，先让一个**基于用户证据的影子读者**做预审。对外统一叫“用户偏好代理”或“影子用户”；不得宣传成“完全像你的数字分身”、人格复制或能百分之百替用户做决定。内部实现可使用 `UserPreferenceProxy` 和 `BookReaderProxy`，但普通界面不必暴露英文名。

首版采用非参数化路线：从证据记忆中检索当前相关偏好，对候选做比较和排序，以盲读方式预审正文，根据真实接受、拒绝和纠正继续校准，并在证据不足或相互矛盾时主动说“我还判断不准”。首版不做用户专属权重蒸馏、LoRA 或微调；未来只有留出集证明轻量排序器或偏好模型比现有方案稳定有效时，才进入评估，不能提前承诺。

偏好严格分成三层：

- **长期跨项目偏好**：多本书、多次明确证据支持的稳定倾向；
- **本书偏好**：只服务当前小说的题材、人物、节奏和表达选择；
- **当前卷 / 当前章节临时意图**：本次场景、当前情绪或局部实验，不得污染前两层。

每条重要偏好都要能追溯用户原话或行为证据、当前范围、支持与反证、置信度、版本、可撤销状态和是否由用户确认。学习来源可以是用户明确表达、候选选择、拒绝后的诊断确认、最终通过版本、用户有权使用资料的抽象特征以及交互习惯；但“上传过”“读过”都不等于“喜欢”，仓颉自己的生成和判断也不能循环强化成用户金标准。

影子用户只能预测、排序、预审、指出可能偏移或建议暂停。它不能替用户正式通过章节、把内容写进“已经确定”的故事记忆、决定重大剧情，也不能覆盖用户真实反馈或绕过费用、安全、权限、版本和恢复边界。

这项能力按 P0–P5 逐步落地，不允许先做一个会模仿口吻的演示再补证据治理：

- **P0 事件 / 证据数据基础**：先记录明确表达、选择、拒绝诊断、最终通过版本、授权资料抽象特征和交互习惯，并支持三层范围、支持/反证、版本和撤销；
- **P1 被动画像**：让用户能看见、纠正和撤销仓颉当前理解，但不让画像自动替用户做决定；
- **P2 影子预审**：比较候选、预测接受/拒绝、盲读章节，并在判断不准时主动弃权；
- **P3 连续生成防偏**：接入章节前计划门、章后双审校、累计漂移、黄色缩小窗口和红色安全暂停；
- **P4 真实反馈校准**：只用真实用户反馈检查预测、排序、弃权、漏报和误报；
- **P5 轻量模型评估**：只有独立留出集和真实用户抽样证明稳定增益时才评估轻量排序器/偏好模型，不是首版默认能力。

论文和外部研究只帮助选择方法，不能把其实验数字、胜率或“像用户”的指标直接变成仓颉宣传承诺。

## 7. 正文阅读与自由反馈

### 7.1 阅读页

正文首先必须像小说，不能像标注数据集，也不能因为提供了编辑能力就变成编辑器优先产品。横屏使用共同创作工作台，竖屏使用单焦点阅读；两者共享同一章节、版本、滚动位置和选区状态。默认校准靠“选中或引用内容 → 用大白话告诉仓颉 → 仓颉追问或执行修改”，手动逐字编辑不是必经步骤。

### 7.2 第一层选区菜单

用户长按并拖动任意字句后，第一层菜单只保留最自然、最通用的动作：

```text
复制 | 问仓颉 | 更多
```

**选区只表示“我现在想讨论这里”或“从这里开始看问题”，不自动表示锁定、不自动表示喜欢，也不能据此推断用户喜欢或不喜欢的原因。** 选择同一段文字后，用户可能想提问、补充画面、指出问题、讨论前后因果或发起修改；系统必须等待后续语言或明确动作确定语义。

“更多”中可以提供：

- **这段我喜欢**：保存软性的正向偏好证据；
- **这个感觉别丢**：要求后续保留氛围、效果或作用，但不锁死当前字句；
- **只讨论这段**：限制当前对话焦点，不自动限制最终修改影响范围；
- **标记为问题**：保存待诊断的负向证据，原因可以为空；
- **锁定文字不变**：极少使用的精确硬保护；
- 补充备注和系统原生的全选等动作。

只有用户明确点击“锁定文字不变”，或明确说“这句一个字都不要动”“这几个字必须原样留下”等无歧义命令，才建立绑定章节版本和精确字节范围的硬锁定。普通选区、“这段我喜欢”和“这个感觉别丢”都不能静默升级为硬锁定。

仓颉可以根据上下文提出“你可能喜欢的是这种克制感”之类的**可纠正推测**，但必须标成推测，并允许用户直接说“不对，我喜欢的是他没解释”来修正；推测不能冒充已确认偏好。

### 7.3 选中文字后问仓颉

主路径：

1. 用户自由选择半句话、一句话或跨段内容；
2. 点击“问仓颉”；
3. 横屏打开或切换右侧“仓颉”，竖屏切换到单焦点“仓颉”；
4. App 自动插入引用卡，至少携带章节、正文版本、精确选区和必要前后文；
5. 用户只需说“这里不对”“太尬了”“我想换个画面”“我说不清”或直接描述新想法；
6. 仓颉能从选区和上下文直接理解时，不强迫用户分类或答问卷，先形成大白话“修改影响范围预览”；
7. 只有确实存在多种合理解释时，才进行动态追问、对比选择或短试写验证。

引用卡必须绑定原始章节和版本，正文变化后能够识别过期选区并重新定位或请求用户确认，不能悄悄改到错误位置。引用选区是诊断和修改的起点，不是系统可以直接覆盖的最终边界。

### 7.4 动作语义

| 动作 | 用户理解 | 系统行为 |
|---|---|---|
| 选择文字 | 我现在想让你看这里 | 只建立版本化讨论焦点，不创建偏好、问题标签或锁定 |
| 问仓颉 | 我想让你看着这段理解我的问题或想法 | 自动带章节、版本、精确选区和前后文进入对话 |
| 这段我喜欢 | 这种效果可能值得保留 | 保存软性正向证据；喜欢原因只能先作为可纠正推测 |
| 这个感觉别丢 | 后续修改别丢掉这段的作用或感觉 | 保存软性保留意图，由仓颉总结可纠正的效果描述，不逐字锁死 |
| 只讨论这段 | 当前先围绕这里说 | 限制当前对话焦点；若真实修改会影响后文，仍必须预警 |
| 标记为问题 | 这里不对，但我可能说不清 | 保存负向证据，原因可为空；能理解就预览，不能才追问 |
| 锁定文字不变 | 这些字一个都不能改 | 只有明确动作或明确自然语言命令才建立精确硬锁定 |
| 补充备注 | 我想提供额外画面或要求 | 保存为与选区绑定的用户意图，不自动当成正文 |

### 7.5 修改影响范围预览与依赖重连

**选区是修改起点，不是最终影响边界。** 仓颉在真正改写前，必须判断问题和改动会停留在一句话，还是会扩展到当前场景、整章、后续章节、故事记忆和计划，并特别检查是否触碰已通过正文或用户亲手改过的内容。

普通用户先看到大白话预览：

```text
你选中的是林越回答师父的这句话。

我判断只改这句话会让后面的动作仍沿用旧情绪，读起来可能断掉。
准备直接修改：这句话和紧接着的两个动作。
可能连带修改：本章结尾对师父态度的回收，以及第二章开头的一句回想。
保持不动：你亲手改过的雨夜描写；已经通过的第一章开场。

[连带改顺后面]
[只改这里但可能不连贯]
[另建版本试试]
[先别改]
```

影响判断至少覆盖：

- 当前句子和相邻段落的语义、视角与语气；
- 当前场景的动机、动作、信息顺序和结果；
- 本章后续场景与章节结尾；
- 后续章节仍在使用的旧事实、人物态度、因果和承诺；
- 已通过正文、用户人工改稿、明确硬锁定和正在使用的工作内容；
- 故事记忆、人物知识、时间线、计划、线索和题材规则。

“只讨论这段”只限制当前沟通范围，不会让仓颉隐瞒真实依赖；如果用户选择“只改这里但可能不连贯”，系统要保留风险说明和版本记录，不能假装已经完整修好。

尚未通过的工作内容可以按依赖图选择性重生成，只处理真正受影响的句子、场景或章节。已通过内容不能直接覆盖；一旦需要修改，必须保留原版本、建立分支并让用户裁决。用户人工改稿优先作为当前依据，除非用户明确授权替换，AI 不得用旧生成稿覆盖人工文字。

执行时按依赖顺序重连：先修正原因和人物选择，再修场景结果、本章回收和后续引用，最后重新检查人物知道什么、时间顺序、因果链、线索/读者承诺和题材规则。禁止只改中间一处，却让后文继续沿用旧逻辑。

### 7.6 Agent 校准是主流程，手动编辑只是高级兜底

普通用户和熟练作者默认都走同一条 Agent 主路径：

```text
选中或引用正文
→ 用大白话说问题、感觉或新想法
→ 仓颉能理解就直接预览影响，不清楚才追问
→ 用户决定范围
→ Agent 执行修改并重查依赖
```

手动编辑必须**可发现但不打扰**：可以放在阅读页“更多”或次级工具栏中，供确实想亲自改字的用户使用；不能占据默认主按钮，不能主动劝用户逐字润色，也不能因为用户从未进入编辑模式而阻塞前三章校准、章节通过或自动连载。

用户进入手动编辑后：

- 每次编辑会话自动保存为新的正文版本，原 AI 稿和之前版本继续保留，可比较和恢复；
- 人工文字成为当前分析、上下文编译和后续生成的最高优先级依据，旧 AI 稿不得回灌覆盖；
- 人工改稿不自动等于“这章已经通过”，章节仍保持原审批状态，只有明确通过动作才能冻结；
- 输入过程中不逐字、逐句弹影响确认，避免把写作打断成确认劳动；
- 在用户离开编辑、要求仓颉继续修改、继续生成、审批章节或启动后续任务前，集中完成一次影响分析；
- 集中分析人物知道什么、时间、因果、线索/读者承诺、题材规则、后续依赖和已通过内容；触碰已通过正文时仍必须保留旧分支并请用户裁决。

## 8. 仓颉的自主执行与确认边界

### 8.1 默认是“关键事情问我”

普通用户不需要先学习权限等级。仓颉默认遵循一句话：**能安全撤回、不会改变整本书方向的日常创作决定直接做并告诉我；真正会改写作品命运的大事，只有在我授权的范围内才替我决定，否则先安全停下来。**

设置中可以提供：

- **少打扰我**：普通决定直接做；用户还可以按决定类别、某一卷或某些章节授权仓颉代为处理指定重大变化；
- **关键事情问我**：默认；正文、重要方向和未被授权覆盖的重大变化按不同层级处理；
- **每一步都让我确认**：适合希望完全手动掌控的用户。

切换模式或授予创作权限只改变创作决定由谁来做，不能关闭费用硬上限、任务完整性、安全与工具权限、版本/幂等/checkpoint、不可逆删除保护或向外部服务发送数据前的授权。

### 8.2 安全可逆的事情直接做

创建小说、保存念头、整理未确认内容、查询进度和费用、运行检查、保存恢复位置、暂停或恢复任务、生成尚未提交的草稿，可以直接执行并在完成后简单告知。

用户说“先停一下”时，仓颉应直接在安全位置暂停：

> 已经暂停，正文和进度都保存好了。随时可以继续。

禁止再弹“是否确认暂停”，也禁止只在文字上声称暂停而没有真实工具回执。

### 8.3 正文和重要方向先给结果，再确认

完整章节、用户已经看过的正文修改、作品核心方向、重要人物或世界规则、从旧章建立的新分支，必须先展示用户真正关心的内容、影响和必要费用，再由用户决定。前三章处于校准期，每章都必须明确确认。

章节结果页不以复杂审批表为主。正文阅读器和仓颉对话只提供轻量“就按这版继续 / 和仓颉聊聊”，同时理解“可以”“继续下一章”“按这个感觉往下写”等语义明确的自然语言为本章批准。只有明确批准才触发冻结；“还行”“差不多”等含糊肯定不能擅自通过，仓颉只追问一次：“你想继续调一调，还是就按这版进入下一章？”

每章明确通过后，Typed Tool 才按固定顺序保存精确冻结版本、结算故事记忆/人物知识/线索与承诺、写入 checkpoint，再解锁下一章。第三章通过只让连续创作变得可申请，不自动开始连载。

### 8.4 分级创作授权与重大变化暂停

普通、可逆且不改变整本书方向的决定，例如场景中的小动作、过渡方式、非核心配角的临时反应和可撤销的局部安排，由仓颉直接执行并用一句大白话告知，不反复请求确认。

会改变主角核心目标、重要人物生死/永久背叛/彻底黑化、核心关系、世界或能力硬规则、主线、卷纲、结局承诺，或造成大量已通过内容失去依据的决定属于重大创作变化。仓颉先检查当前是否有有效授权覆盖这个**决定类别和作品范围**：

- 已覆盖：仓颉可以代为决定，但执行后必须醒目标记“我刚刚替你决定了一件会影响后续的大事”，说明实际选择、影响章节和可回看的授权；
- 未覆盖：必须在安全 checkpoint 前暂停，不能先写完再补问，也不能只抛一句“请决定”。

暂停卡必须同时给出为什么暂停、会影响哪些内容、2–3 个具体方向、仓颉推荐及理由，并且一次只问一个容易回答的问题：

> 这里要决定师父是否永久背叛。它会改变主角接下来一卷的目标，也会影响第三到第七章已经埋下的信任线。
>
> 方向一：师父真背叛，冲击最强，但整卷会转成复仇。
> 方向二：师父假背叛，我更推荐；既保留冲击，也不会丢掉你喜欢的师徒感。
> 方向三：先制造误会，把真相留到卷末。
>
> 你更想保住师徒感，还是接受主线转成复仇？

用户可以直接说“这一卷的人物生死你替我决定”“第 20–30 章的能力代价你自己拿主意”“核心感情线都先问我”。授权必须可查看、可撤销、版本化并记录来源、类别和小说/卷/章节范围；一次选择不得静默变成永久授权。撤销只影响尚未执行的决定，已执行重大决定仍保留记录并进入正常版本治理。

费用硬上限、任务完整性、工具权限、安全边界、外部数据披露和版本/幂等/checkpoint 不能被任何创作授权或“少打扰我”模式绕过。

### 8.5 受治理的修改影响预览

所有对用户已看到正文的实际改写都必须经过受治理的影响预览和授权。普通用户看到选中的起点、真正需要改动的范围、会受影响的后文、人工改稿和已通过内容保护、可选处理方式以及必要费用，不看到工程术语。

```text
我已经从你选中的位置检查了前后依赖。

直接修改：
· 第一章中间这句话和后面的两个动作；
· 让林越表面克制，但身体反应暴露真实情绪。

可能连带：
· 本章结尾对师父态度的回收；
· 第二章开头仍沿用旧情绪的一句回想。

优先保留：
· 你亲手改过的雨夜描写；
· 已经通过的开场版本；
· 你明确锁定不变的文字。

预计花费约 1～2 元。

[连带改顺后面] [只改这里但可能不连贯]
[另建版本试试] [先别改]
```

用户确认后，Typed Tool 才能基于绑定章节版本、依赖影响、人工改稿来源、明确硬锁定、审批范围和幂等键提交修改。工作内容允许选择性重生成；触碰已通过正文时必须改走分支和用户裁决，不能直接覆盖。默认不出现 Rewrite Scope、Artifact Revision、Context Hash、Binding 或 Receipt，但后台治理不得削弱。
## 9. 从一个念头到小说的完整体验

### 9.1 一个念头

用户：

> 我想写一个主角能看见别人死亡倒计时的小说。

仓颉：

> 这个念头最有意思的地方，可能不是“看见死亡”，而是明知道一个人快死了，却不知道该不该救。我先确认一种感觉：你更想看主角利用能力改变命运，还是不断发现有些死亡根本改变不了？

### 9.2 用户说不知道

仓颉改用两个短开场，而不是要求用户给主线和爽点。探索期可以随时生成 100–300 字画面、微型试写、候选开场、能力代价或章节结尾候选，无需先完成完整策划、建立正式开篇基础或逐字段审批。这些结果只是可撤销的探索材料，不等于完整第一章，也不会静默变成用户已经确认的设定。

### 9.3 人话总结

```text
我目前对这个念头的理解

大概故事：……
你想要的感觉：……
已经排除：……
还没确定：……

[对，就按这个继续]
[大体对，有地方要改]
[不对，重新理解]
[我也不知道，先试写看看]
```

### 9.4 到需要时再建立小说并开始第一章

前面的对话已经自动保存，但不会因为一句念头就产生空书。当用户明确继续、形成首个长期正式成果、需要故事记忆或准备开始正文时，仓颉通过 Typed Tool 无表单建立小说，并用大白话告知。

完整第一章开始前，普通界面只展示一张结果，不要求填写设定表或逐字段审批：

```text
我准备这样写

故事感觉：紧张、诡异，但主角不是冷血旁观者。
主角现在：第一次看见身边人的死亡倒计时，还不知道能力为什么出现。
第一章会发生：他试着救下一个只剩十分钟的人，却发现改变死亡需要付出代价。
结尾得到：读者看见第一次救人结果，并知道这个能力不是白送的。
明确避免：不一次讲完能力原理，不混入都市科技解释。
还没定死：能力代价的长期形式，先按可撤销的临时假设写。

[就这样开始]
[再聊聊]
```

前台不出现“制作圣经”“CreativeContract”“正典”“开篇合同”或“章节节拍”等术语。用户点击“就这样开始”，在对话中说“开始写第一章”“你替我决定”“直接写”或其他语义明确的开始指令，都构成授权；尚未决定的内容保持可撤销临时假设，不得静默升级成用户确认。

### 9.5 生成期间

```text
正在做：生成《逆命登仙》第一章正文
接下来：检查人物状态并保存安全位置
需要你：无

上次安全保存：第一章场景 2
预计本轮：约 3～6 元
已经用掉：1.8 元
```

用户可以直接问：“现在到哪了？”“为什么停了？”“还要多少钱？”“之后还能继续吗？”仓颉每次都从真实任务状态源查询，不根据聊天记录猜测。

用户说“写完这一章暂停”时，当前章先完成审校、临时记忆结算和 checkpoint 再停；用户说“现在暂停”时，立即取消当前请求并只保留最近完整 checkpoint 与残缺临时内容；说“到这里就结束，刚才的留着”时，停止任务但保留未采用成果；只有用户明确要求放弃，并看过影响说明后，才处理未采用成果。三者不得共用一个含糊的“取消”动作。

网络断开、App 挂起、模型繁忙、预算硬上限或重大故事分歧发生时，任务在安全位置暂停，并在对话、“这次结果”和“AI 任务”页显示相同的人话原因、保存位置和恢复动作。同一时间只运行一个主要创作任务，新的生成请求进入队列或先询问。

完整第一章在后台仍运行生产级流水线：

```text
计划
→ 研究覆盖检查
→ 写作
→ 人物知识检查
→ 连续性检查
→ 题材纯度检查
→ AI 味检查
→ 有限修正
→ checkpoint
→ 第一章结果
```

总编剧在后台形成开篇基础、题材研究、前三章承诺、人物与世界规则、用户偏好、关键未知和临时假设边界，但不把这些专业结构变成用户表单。检查和修正有明确轮次与预算上限，禁止无限修订。

第一章生成后状态只是“供你看看”。它可以进入阅读器和“这次结果”，但不会自动冻结，也不会进入故事记忆的“已经确定”。

### 9.6 阅读、反馈、修改

用户像读者一样阅读。哪里不符合预期，就自由选中文字并点击“问仓颉”；App 自动带上章节、版本、精确选区和前后文。仓颉能理解时直接给修改影响范围预览与局部结果，不清楚时才动态追问。整章“不对味”也允许，不要求先给专业原因。

### 9.7 确认章节

章节结果直接留在连续阅读器中，右侧仓颉提供最少量操作：

```text
[就按这版继续] [和仓颉聊聊]
```

用户也可以直接说“可以”“继续下一章”“按这个感觉往下写”。这些明确表达与按钮具有相同的本章批准语义。若用户只说“还行”“差不多”，仓颉不能猜测已经通过，只问一次：“你想继续调一调，还是就按这版进入下一章？”

只有用户明确通过第一章后，仓颉才通过 Typed Tool 保存并冻结当前精确版本，结算人物当前状态、世界规则、人物知识、线索与承诺，写入 checkpoint，然后准备第二章。生成完成、打开阅读、提出修改或含糊肯定都不等于通过。

```text
第一章已经确定，进度也保存好了。
以后继续写时，我会把这一章发生过的事情当成既定事实，
不会让人物随便忘记，也不会让设定前后矛盾。
接下来我会准备第二章。
```

第二章和第三章重复同一逐章确认合同。

### 9.8 三章后持续创作

第三章明确通过后，仓颉仍然停在可阅读、可修改状态，不自动生成第四章。仓颉先用大白话说明：连续创作会在授权后自动推进、最多领先多少章、费用怎样受预算控制；普通可逆决定会直接处理，未被用户分级授权覆盖的人物生死、主线换轨、世界规则变化等重大决定会在安全保存后暂停。然后只申请一次连续创作授权：

```text
前三章已经确定，进度也保存好了。
我现在对你喜欢的节奏、人物感觉和表达方式有了初步了解。

如果开始连续创作，我会按已经确认的感觉继续往下写。
默认一次连续准备 3 章；你随时可以说改成 1–5 章，但最多只保留 5 章你还没读过的领先版本。
普通、可撤回的推进我会直接处理并告诉你。
遇到会改变故事方向的大事，我会先看你有没有授权我处理这一类决定；没有的话会保存进度，给你具体方向和我的推荐后再问。
费用、安全、权限和向外发送资料的边界，无论你给我多少创作自主权都不会绕过。
开始以后，普通章节不会每章都机械问一次。

[开始连续创作]
[先保持现在这样]
```

用户说“开始连续写”“按这个感觉往下写吧”等明确自然语言同样构成连续创作授权。默认连续准备 3 章，用户可以直接说“这次只写一章”“以后每次准备五章”，或在设置中调整为 1–5 章；首版任何时候最多保持 5 章尚未阅读的领先版本。未授权时小说保持可阅读、可修改，不启动后台连载；授权后持续推进，直到用户暂停或撤销连续创作授权、达到硬边界，或出现未被当前分级创作授权覆盖的重大变化。连续创作授权不自动等于“所有重大决定都交给仓颉”，后者必须按类别和范围单独记录。

自动章节必须逐章生成：第 N 章正文完成后，先完成审校、把本章事实结算为临时故事记忆并保存 checkpoint，才能开始第 N+1 章。未读章节统一标记为“仓颉准备的版本，等你看”，可以作为后续工作上下文，但不等于用户确认，也不能进入“已经确定”。前章被修改时保留旧分支、做影响分析并只重生成真正受影响的工作内容。

用户说“写完这一章暂停”时，当前章在安全 checkpoint 完整收尾后停止；说“现在暂停”时立即取消当前请求，未完成输出只作为可丢弃的临时内容。恢复必须复用任务、章节、幂等键、用量对账和最近 checkpoint，不得重复正文或重复扣费。
连续生成的每一章还要经过独立防偏流水线：

```text
章节前计划门
→ 正文生成
→ 独立硬规则与连续性审校
→ 影子用户盲读预审
→ 累计漂移检测
→ 正常继续 / 黄色缩小窗口 / 红色安全暂停
```

章节前先确认本章目标、当前卷承诺和本章不能越过的边界；章节后由全局故事审校器独立检查人物知识、时间、因果、世界规则、题材纯度和未兑现承诺。影子用户与全局故事审校器必须分开，盲读时不能先看到后者的结论，避免顺着“标准答案”假装懂用户。黄色表示连续生成仍可继续，但应减少领先章节、缩小自动化窗口并更早请用户看；红色表示在安全 checkpoint 后暂停。影子用户只能提供风险和建议，最终继续、暂停、章节批准、故事记忆合并和重大剧情裁决仍走原有受治理权限。

### 9.9 自动连载中的对话控制

用户：

> 现在怎么样了？

仓颉：

> 第四章已经写完，正在检查人物前后是否一致。第五章完成了章节安排，还没开始正文。当前共花费 8.4 元。有一件会改变师徒关系的大事没有在你的授权范围里，所以我已经保存进度并暂停。下面有三个具体方向和我的推荐，你只要回答一个简单问题。

## 10. 术语翻译契约

| 内部术语 | 默认用户文案 |
|---|---|
| Creative Contract / Bible | 这本书目前确定的方向 |
| Canon | 已经确定、以后不能随便矛盾的设定 |
| Working Canon | AI 暂时按这个继续写的内容 |
| Artifact | 普通界面不显示这个词，直接显示具体结果名称 |
| Approval | 就按这个继续吗 |
| Frozen Chapter | 这章确定了 |
| Rewrite Scope | 这次准备改哪些地方 |
| Tool Receipt | 仓颉刚才实际做了什么 |
| Checkpoint | 已经保存的恢复位置 |
| Branch | 另一个版本 |
| Promise Ledger | 后面不能忘的事 |
| Character Knowledge | 这个人物现在知道 / 还不知道 / 错误地以为什么 |
| Story Memory Projection | 故事记忆 |
| Reconciliation | 正在确认刚才是否真的完成 |
| Task Run | AI 任务 |
| Safe Pause | 已经暂停，进度已保存 |
| Stop and Keep Results | 已停止，刚才生成的内容还保留着 |
| Discard Results | 放弃这些未采用内容 |

所有用户文案先写大白话；专业术语只在技术详情或高级设置中出现，并附解释。

## 11. 禁止的设计

1. 默认打开创建小说表单；
2. 一次问主线、爽点、人物弧、世界观和结局；
3. 固定三个问题无论回答什么都继续；
4. 不满意必须先填写原因或选择专业分类；
5. 把“喜欢 / 不喜欢”放成每次正文反馈的强制第一步；
6. 正文每段一个框、一个 Lock 按钮；
7. 审批卡确认后不关闭；
8. 横屏增加第四列分别展示正文、仓颉、结果和导航；
9. 横屏或竖屏用省略号隐藏关键决定；
10. 把横屏工作台简单缩窄成竖屏多栏；
11. Activity Bar 常驻文字挤压正文，或只有图标却没有长按说明和无障碍名称；
12. 默认显示 Hash、Revision、Binding、Receipt；
13. Agent 只口头说执行，后台没有真实工具结果；
14. 让用户手动选择哪个 Agent、提示词、上下文或审校轮次；
15. 一次生成多个完整方案，把判断负担推给用户；
16. 用 Keychain、Candidate Set、Probe 等诊断页面代表产品主页；
17. 把“第二次覆盖安装”当成正常升级步骤；
18. AI 静默决定人物死亡、主线换轨、大范围重写或预算越界；
19. 使用纸墨朱红、卷轴、毛笔、印章等仿古主题冒充“仓颉”品牌；
20. 声称仓颉使用 Claude 官方色值、官方组件或官方品牌资产；
21. 每保存一句对话就自动创建“暂未命名”的空书；
22. 把普通追问、寒暄或每条模型消息都复制成“这次结果”卡片；
23. 要求用户手工复制产物到章节、故事记忆、资料、任务或创作记录；
24. 在普通界面暴露 Artifact、CanonFact、Revision Hash、Tool Receipt 等内部名词；
25. AI 推测未经确认就升级成用户决定或已确定设定；
26. 浏览书架、打开书籍详情或只阅读正文时偷偷切换当前创作上下文；
27. 书架条目展示项目 ID、内部状态、数据库字段等技术信息，或点击书籍后替换中央对话；
28. 把故事记忆做成必须由用户填写和维护的设定表；
29. 在故事记忆中暴露 Canon、TruthScope、PromiseLedger、版本哈希等后台结构，或把 AI 推测显示成“已经确定”；
30. 修改一条会冲突大量已通过正文的记忆时静默覆盖旧事实，不先展示影响和受治理修改方案；
31. 把研究设计成只能由用户主动点击的搜索框，或每遇到一个名词就盲目联网；
32. 只根据 LLM 自报置信度决定是否研究，或把题材知识包和外部资料直接当成本书已确定事实；
33. 用户关闭联网或限定本地资料后仍偷偷联网；
34. 探索期要求先填写完整策划、制作圣经或逐字段表格，才允许生成 100–300 字小样；
35. 完整第一章前展示多张专业字段审批卡，而不是一张“我准备这样写”；
36. 把“直接写”“你替我决定”等明确自然语言授权判成无效，或把未定内容静默升级为用户确认；
37. 第一章一生成就自动冻结、写入“已经确定”故事记忆，或不经用户通过就作为后续既定事实；
38. 把选中文字自动解释为喜欢、锁定或不满意原因，或把“原样保留”继续放在第一层主路径；
39. 把选区当成最终修改边界，只改中间一句却让场景、章节或后文继续沿用旧逻辑；
40. 用 AI 旧稿覆盖用户人工改稿，或不建分支就直接覆盖已通过正文；
41. 把手动逐字编辑设计成普通用户或熟练作者的默认校准主路径，或因为用户不手改而阻塞创作；
42. 用户手动改一个字就弹一次影响确认，或在编辑会话尚未结束时用连续弹窗打断输入；
43. 把人工改稿直接标成“这章已经通过”，或覆盖删除原 AI 稿和旧版本；
44. 把用户上传的小说、片段或个人作品宣传成“训练了专属模型”或“微调了模型权重”；
45. 从参考资料中复刻具体表达、长段落、独特桥段，或保存成可复现的版权正文语料；
46. 把一次选择或一次对话中的 AI 推测静默升级为永久、跨项目偏好；
47. 用户只说“这章不对劲 / 我说不上来”时弹出不满意原因表格、专业分类或要求用户先自我诊断；
48. 在没有达到可行动清晰度、没有反映当前理解和没有展示影响范围前，直接换提示词盲目整章重抽；
49. 把内部诊断给出的候选原因、小样反应或 AI 推测静默升级成用户确认事实、永久偏好或已确定故事记忆；
50. 把章节结果做成复杂审批表，或只认按钮点击、不理解明确自然语言批准；
51. 把“还行 / 差不多”等含糊肯定静默当成章节通过，不做一次澄清；
52. 第三章一通过就自动开始第四章或连续创作，没有单独的一次授权；
53. 已获得连续创作授权后，普通章节仍机械逐章弹审批，或未授权时偷偷在后台生成；
54. 把“用户偏好代理 / 影子用户”宣传成完全复刻用户的人格数字分身，或承诺它能百分之百代替用户判断；
55. 首版暗中训练用户专属权重、做 LoRA / 蒸馏，或把未来可能研究的轻量模型写成当前能力；
56. 把长期跨项目、本书和当前卷 / 章节临时偏好混在一起，或缺少原始证据、支持/反证、范围、版本、撤销与确认状态；
57. 把仓颉自己生成的正文、评价或影子用户预测反向当成用户金标准，形成自我强化闭环；
58. 因为用户上传或读过某部作品就自动判定喜欢，或从参考资料复刻版权表达、长段落和独特桥段；
59. 让影子用户正式批准章节、合并已确定故事记忆、决定重大剧情或覆盖用户明确反馈；
60. 让用户偏好预审读取全局故事审校结论后再假装盲读，或只看单章、不检测连续多章累计漂移；
61. 用论文实验数字、胜率或榜单直接作为仓颉的宣传承诺和验收目标；
62. 上传材料后就要求用户选择付费模型，或在基础阅读和搜索可用前阻塞等待整本深度分析；
63. 在未展示发送范围、不会发送什么、Provider/模型、用途、预计费用和预算上限时，静默把材料发给外部服务；
64. 把一次联网授权静默扩张到新的文件、章节、用途、Provider/模型或预算，或用户撤销后仍继续外发；
65. 每次新增、修改、暂停或恢复都重新分析整本书，造成重复外发、重复计算和重复扣费；
66. 所有资料固定用同一理解器，或只按 ZIP 整包分类而忽略内部文件和混合片段；
67. 让参考资料自动成为本书设定，让参考小说作为事实来源，或绕过项目、资料类型、用途、确认状态、权限和外发授权隔离；
68. 共享 Evidence Index 时丢失不可变原文、版本、定位、哈希、增量 checkpoint 或结论回到原文的证据链。

## 12. 设计定调验收

后续每一阶段都必须通过以下五个问题：

1. 一个只会看小说的人，不看说明能否知道下一步做什么？
2. 用户是否只需要说人话、看正文和表达感觉？
3. Agent 是否真的能操作软件，而不是只有聊天外观？
4. 专业复杂度是否仍然完整存在，但默认隐藏？
5. 横屏是否保持正文约 2/3、右侧约 1/3且右侧只用“仓颉 / 这次结果”切换，竖屏是否保持单焦点？
6. 视觉是否克制、安静、现代，并明确属于仓颉自己的中性暖色设计系统？
7. 默认是否做到“普通可逆决定直接做并告知、重大变化按类别和范围检查授权、未覆盖才安全暂停、硬边界永不绕过”？
8. 用户真正重要的决定是否仍由用户掌握？
9. 对话是否自动保存但不会制造空书，并能在真正需要时由仓颉无表单建书？
10. 动态意图是否做到“一次一个关键问题、很快给可见小样、该停时就停”，且推测不会冒充确认？
11. “这次结果”是否只收集真实产物，普通闲聊不生成卡片，采用后能自动写入正确位置并继续追溯？
12. 普通界面是否只使用有限大白话状态，而不暴露 Artifact、CanonFact、Revision Hash、Tool Receipt？
13. 小说图标是否只打开左侧书架，书籍详情是否只在左侧 push，中央对话、草稿、流式和阅读位置是否保持？
14. 书架是否只显示标题、大白话进度和最近时间，浏览/阅读是否不会偷换创作上下文，只有四种明确动作才绑定并提示？
15. 仓颉是否会按固定知识顺序主动判断研究缺口，尊重关闭联网、本地限定和研究预算，并把外部资料隔离为无权限的不可信参考？
16. 探索期是否无需完整策划即可给 100–300 字小样，完整第一章前是否只展示一张“我准备这样写”，并接受按钮或自然语言启动？
17. 第一章生成后是否只标“供你看看”，只有用户通过后才冻结正文并结算故事记忆、人物、世界、线索和下一章？
18. 选区第一层是否只有“复制 / 问仓颉 / 更多”，且选区本身不会自动变成喜欢、问题或硬锁定，只有明确命令才锁定文字？
19. 修改前是否会展示句子、场景、章节、后文、已通过内容和人工改稿的真实影响，并提供“连带改顺后面 / 只改这里但可能不连贯 / 另建版本试试”？
20. 工作内容是否选择性重生成，已通过内容是否必须分支裁决，人工改稿是否优先，修改后是否重查人物知识、时间、因果、承诺和题材规则？
21. 不进入手动编辑模式时，普通用户和熟练作者是否都能仅靠选区/引用、对话和 Agent 完成完整校准？
22. 手动编辑是否只是可发现的次级能力，并且新版本自动保存、旧 AI 稿可恢复、人工文字优先但不会自动通过章节？
23. 手动编辑时是否避免逐字弹窗，并只在离开编辑、继续修改/生成、审批或启动后续任务前集中做影响分析？
24. 用户授权参考资料后，是否只生成带来源、可解释、可确认/撤销且可设置本书/跨项目范围的抽象参考画像，而不复刻版权表达？
25. 交互偏好是否区分一次性、本书和跨项目范围，AI 推测是否与用户确认分层，一次选择是否不会自动变成永久偏好？
26. 普通界面是否明确说明“学习不会训练或微调模型权重”，首版只使用本地可审阅记忆、检索和上下文编译？
27. 用户只说“这章不对劲 / 我说不上来”时，仓颉是否先基于对话、偏好、故事记忆和正文给 2–3 个大白话候选原因，一次只问一个容易回答的问题，必要时用可撤销小样验证，并在达到可行动清晰度后先反映理解、展示影响范围再修改？
28. 模糊拒绝过程中是否没有原因表格、专业分类和盲目整章重抽，且所有诊断候选都保持为 AI 推测，只有用户确认后才进入确认偏好或正式修改依据？
29. 前三章是否逐章明确确认，章节结果是否以阅读器和仓颉对话中的“就按这版继续 / 和仓颉聊聊”为主，并接受明确自然语言批准？
30. “还行 / 差不多”等含糊肯定是否只触发一次“继续校准还是按此继续”的澄清，而不会冻结章节？
31. 每章通过是否严格按“冻结精确版本→结算故事记忆/人物知识/线索→checkpoint→下一章”执行并可追溯？
32. 第三章通过后是否仍停在可阅读/修改状态，只申请一次连续创作授权；授权后普通章节不再机械逐章询问，未授权则绝不自动连载？
33. 自动连载中的普通可逆决定是否直接执行并告知，而未被有效范围授权覆盖的主角目标、重要人物生死/永久背叛/彻底黑化、核心关系、硬规则、主线/卷纲/结局承诺变化是否在安全 checkpoint 前暂停？
34. 重大变化暂停卡是否说明原因与影响、给出 2–3 个具体方向和仓颉推荐，并一次只问一个容易回答的问题，而不是抛空白决定？
35. 用户是否能用自然语言按类别、卷或章节查看、授予和撤销创作代决策权限；一次选择是否不会静默升级成永久授权；已授权重大决定执行后是否醒目标记？
36. 费用硬上限、任务完整性、工具权限、安全、外部数据披露以及版本/幂等/checkpoint 是否在任何自主级别下都不可绕过？
37. 连续创作是否默认准备 3 章，且用户能用人话或设置调整为 1–5 章，首版最多保留 5 章尚未阅读的领先版本？
38. 正文是否严格逐章、单 Writer owner 生成，并在每章审校、临时故事记忆结算和 checkpoint 后才开始下一章；并行研究/审校是否从不争夺正文写权限？
39. 未读自动章节是否只标“仓颉准备的版本，等你看”，可作工作上下文但不冒充用户确认？
40. 前章修改是否保留旧分支、先做影响分析并只重生成受影响内容？
41. “写完这一章暂停”和“现在暂停”是否具有不同真实行为；立即暂停后的残缺输出是否只作临时内容？
42. 任何恢复是否按任务、章节、幂等键、用量对账和 checkpoint 继续，绝不重复生成或重复扣费？
43. 对外是否只使用“用户偏好代理 / 影子用户”，且明确不承诺完全像用户的数字分身或人格复制？
44. 首版是否只使用基于证据的非参数化偏好记忆、检索、候选比较、盲读预审、真实反馈校准和主动弃权，而没有用户专属权重蒸馏、LoRA 或微调？
45. 长期跨项目、本书、当前卷 / 章节临时意图是否严格隔离，每条偏好是否带原始证据、支持/反证、范围、置信度、版本、撤销状态和用户确认状态？
46. 上传或阅读是否都不会自动变成“喜欢”，仓颉自己的生成与判断是否绝不会反向强化成用户金标准？
47. 影子用户是否只能预测、排序、预审和建议暂停，不能正式批准章节、合并已确定故事记忆、决定重大剧情或覆盖用户反馈？
48. 连续生成是否真实经过章节前计划门、独立硬规则/连续性审校、影子用户盲读预审和累计漂移检测，并确保偏好代理看不到全局审校结论？
49. 黄色漂移是否会减少领先章节、缩小自动化窗口并提前反馈；红色漂移是否会在安全 checkpoint 后暂停，且两者都不能绕过原有预算、权限、版本和重大决定治理？
50. 偏好代理是否会在证据不足或冲突时合理弃权，评测是否同时检查接受/拒绝预测、候选排序、校准、误报/漏报、自动化覆盖和真实用户抽样，而不是只听 LLM Judge？
51. 论文与研究结果是否只作为方法参考，不把实验数字直接变成产品承诺？
52. 导入资料、正文、人工改稿、研究和授权参考资料是否保留不可变原文版本，任何摘要、向量、抽取或后续修改都不能覆盖，重要结论能否打开原文位置？
53. 仓颉是否按任务自适应组合全文、向量、章节顺序和人物/事件/认知/伏笔关系检索，而不是每次全书扫描、只用关键词或只用向量？
54. 当前证据不足时是否自动从场景扩大到章节、当前卷、相关关系、全书和必要资料，并记录扩大原因；仍不足时是否诚实显示“暂时无法确认”？
55. 人物认知、事件发生、时间因果、能力/资源、伏笔、已通过正文、人工改稿和研究依据等重要结论，是否都回到原文复核，而不是由 LLM 单独断言？
56. 索引是否先让原文可读可搜，再渐进建立人物、事件、认知、状态、伏笔、向量和关系；中断能否恢复，覆盖不足时是否明确显示整理到哪里？
57. 上传参考小说是否只生成带来源的抽象写法候选，并在用户确认后才进入偏好；上传/阅读是否不等于喜欢，且没有复刻具体表达、长段落、独特桥段或版权正文？
58. 首版是否仅使用 SQLite/GRDB、FTS5、轻量向量和结构化关系表，且没有引入 Neo4j、Qdrant、完整 GraphRAG/LightRAG 服务或云端知识图谱硬依赖？
59. 上传材料后是否先自动完成不调用付费模型、不外发内容的本地基础整理，并在原文保存后立即允许阅读和搜索？
60. 首次联网深度理解前是否明确显示发送范围、不会发送什么、Provider/模型、用途、预计费用/预算和增量许可，并且只有用户明确授权后才开始？
61. 授权范围、用途、Provider/模型或预算实质变化时是否重新授权；撤销或选择“只用本地”后是否确实停止外发？
62. 新增或修改材料是否只处理受影响部分；暂停、断网、挂起和失败后是否从安全位置幂等恢复，不重复发送、分析整本书或扣费？
63. 所有材料是否共享不可变原文、版本、定位、哈希、全文/语义检索、增量 checkpoint 和证据回链，同时按类型进入 NarrativeIndex、ResearchIndex、ProjectMaterialIndex 或 PreferenceIndex？
64. 混合 ZIP 是否按文件、必要时按片段分类；无法可靠分类且会影响结果时是否只问一个容易回答的问题，而不是让用户先整理整包资料？
65. 检索是否严格受项目、资料类型、用途、确认状态、权限和外发授权隔离；参考资料是否不会自动成为本书设定，参考小说是否不会被当成事实来源？

如果任一答案为“否”，即使技术功能已经完成，也不能称为产品体验完成。
## Supplementary user-facing contract from 1.md

The following behavior was already approved and is restored here without adding a new setup questionnaire.

### What the user sees when a model is connected

The user selects a concrete Provider, enters a key, retrieves the models available to that key and manually selects the model. Multiple saved connections are allowed, but the current connection is always selected manually. No hidden routing, automatic switching, key polling or failure takeover is visible or permitted.

### Honest driving modes

用户可能接入：

```text
OpenAI
Anthropic
Gemini
DeepSeek
Agnes
自定义OpenAI-compatible接口
其他兼容模型
```

不能假定它们能力完全相同。

仓颉连接 Provider 后要自动进行能力探测：

```text
是否支持流式输出
是否支持标准工具调用
是否支持结构化JSON
是否支持取消请求
是否返回Token用量
是否支持系统提示
是否支持长上下文
是否支持原生搜索
是否支持图像
是否支持Embedding
```

然后给模型分级。

## 完整驾驶模式

模型可靠支持：

```text
工具调用
结构化输出
流式输出
取消
用量信息
```

可以作为仓颉主驾驶员。

---

## 受限驾驶模式

模型支持文本和结构化 JSON，但工具调用不够标准。

仓颉可以通过严格验证的适配器支持部分工具，但限制高风险操作。

例如可以：

```text
查询项目
创建草稿
生成正文
提出工具请求
```

但涉及正式修改和复杂连续任务时，需要更严格验证。

---

## 仅创作模式

模型只能稳定生成文本，无法可靠调用工具。

它可以作为：

```text
正文写手
摘要模型
灵感模型
文风编辑
```

但不能作为控制整个软件的主驾驶员。

仓颉要明确告诉用户：

> 这个模型可以参与写作，但暂时不能可靠地操作项目、任务和审批。你可以保留它作为正文写手，再选择一个支持工具调用的模型负责仓颉对话。

不能让一个不支持工具的模型假装执行成功。

---

### Honest failure at a product gate

例如模型错误调用：

```text
generation.start
```

但当前还没有通过前三章，工具层应直接返回：

```text
拒绝执行
原因：前三章校准尚未完成
当前状态：第一章等待用户确认
可以执行：打开第一章、继续讨论、创建新版本
```

模型再向用户解释：

> 现在还不能开始连续创作，因为第一章还在等你确认。我可以先把第一章打开，或者根据你刚才的意见再调整一次。

因此：

> **模型可以提出行动，但最终执行权属于仓颉工具和状态机。**

驾驶员即使操作失误，高达自身的安全系统也会阻止它撞墙。
