# 仓颉产品体验蓝图

- 状态：产品视觉与交互基准
- 基线：2026-07-18
- 适用设备：11 英寸 iPad Pro，iPadOS 16.6+
- 关联权威方案：`docs/IMPLEMENTATION_PLAN.md`

> 仓颉不是一套让用户学习怎么写小说的工具，而是一位能直接操作整套小说生产系统的创作搭档。

本文回答三个问题：最终产品长什么样、普通用户怎么使用、开发到每一处时应以什么体验为准。

---

## 1. 产品基调

### 1.1 第一感觉

打开仓颉后，用户应该觉得这是一个安静、可靠、会主动帮忙的小说创作 Agent，而不是：

- 专业作者后台；
- 密密麻麻的数据管理软件；
- 小说工程控制台；
- 需要先学习的设定表格；
- 只有聊天外观、实际不会操作软件的普通聊天机器人。

### 1.2 视觉气质

- 暖白或浅灰白背景；
- 接近黑色的深灰正文；
- 低饱和朱红或赤褐作为仓颉强调色；
- 沉稳绿色表示“已经确定”；
- 琥珀色表示“等你决定”；
- 低饱和蓝表示“正在进行”；
- 克制暗红表示“需要注意”；
- 有一点纸张和墨色的东方气质，但不做卷轴、毛笔、印章堆砌；
- 不做闪烁、炫技、赛博朋克式 AI 控制台。

### 1.3 阅读优先

- 正文行距宽松；
- 单行不过长；
- 对话字号略大于普通聊天软件；
- 正文和对话字号可分别调整；
- 工程详情字体更小且默认折叠；
- 动画只用于状态变化、内容出现和页面连续过渡。

## 2. 首次打开

### 2.1 欢迎页

```text
仓颉

有什么想写成小说的念头吗？

你不用会写，也不用先想好主线、人物和世界。
可以告诉我一句话、一幅画面、一种感觉，
甚至只说你最近喜欢看什么。

剩下的，我来陪你想清楚。

[ 我有一个念头 ]  [ 我还没想法 ]

随便说点什么……                         ↑
```

快捷入口只是帮助用户开口，不进入复杂表单。真正第一次安装且没有历史状态时，只显示当前真实可用的开口方式：“接着上次继续”仅在存在历史对话、项目或可恢复任务时显示；“导入已有资料”只有在 S3 基础导入已经真实可用后才显示。不可用入口必须隐藏，不能禁用占位、点击后提示“暂未开放”，也不能以灰色按钮假装功能存在。

### 2.2 非首次打开

根据真实状态显示一条最有价值的继续建议：

```text
晚上好。
《逆命登仙》第一章已经写完，正在等你阅读。

[继续阅读第一章]  [先问问现在的情况]
```

如果有中断任务：

```text
上次在第一章人物一致性检查后安全暂停了。
已经写好的内容没有丢，也没有重复扣费。

[从保存位置继续]  [先看看目前结果]
```

## 3. iPad 总体布局

### 3.1 横屏默认

```text
┌─────────┬────────────────────────────────────────┬──────────────┐
│ 左侧导航 │              中心仓颉对话               │ 结果栏默认收起 │
│         │                                        │              │
│ ＋新对话 │ 仓颉：                                 │ 展开后显示：   │
│ 对话历史 │ 我听下来，你真正想看的不是马上打脸，     │ · 等你决定     │
│ ● 当前对话│ 而是让读者先替主角憋住一口气。           │ · 已经确定     │
│ ○ 昨天   │                                        │ · 正在进行     │
│ 我的小说 │ 我只确认一件事：                       │ · 最近修改     │
│ 阅读修改 │ 第一章结尾更想让他反击，还是发现秘密？   │              │
│ 其他功能 │                                        │              │
│         │ [终于要反击] [发现更大的秘密] [都不对]   │              │
│         ├────────────────────────────────────────┤              │
│         │ 说说你的想法……                  ＋  ↑   │              │
└─────────┴────────────────────────────────────────┴──────────────┘
```

规则：

- 中心对话占最大空间；
- 右侧默认只显示入口和待决定数量；
- 展开后优先覆盖或占约四分之一宽度，不能把对话压成窄条；
- 左侧可收起；
- 左右区域变化不能重建中心对话、丢滚动位置、草稿或正在流式输出的内容。

### 3.2 竖屏默认

```text
┌──────────────────────────────┐
│ ☰   仓颉 ·《逆命登仙》   结果 2 │
├──────────────────────────────┤
│                              │
│ 仓颉：                       │
│ 我大概理解你要的感觉了。       │
│                              │
│ 你不是想让主角马上证明自己，    │
│ 而是想让那些看不起他的人以后    │
│ 发现自己错得离谱。             │
│                              │
│ 我接下来只问一个问题：         │
│ 第一章里，谁最应该在以后后悔？  │
│                              │
│ [退婚的未婚妻]                │
│ [逐他出门的师父]              │
│ [一直欺负他的同门]            │
│ [都不是，我自己说]            │
│                              │
├──────────────────────────────┤
│ 说说你的想法……         ＋  ↑   │
└──────────────────────────────┘
```

规则：

- 竖屏只保留一列主内容；
- 左侧入口从左上角进入独立导航页面；
- 右侧结果从右上角以抽屉或全屏页打开；
- 关闭后回到原对话位置；
- 键盘不能遮挡输入框和发送按钮；
- 长回答和审批内容必须可滚动，不能用省略号丢失关键决定。

## 4. 左侧独立页面

### 4.1 左侧默认页：对话历史和功能入口

左侧默认页不是只有一个模糊的“对话”按钮，而是同时承担对话切换和功能导航：

```text
┌─────────────────────────────┐
│ ＋ 新对话              收起  │
├─────────────────────────────┤
│ 对话历史                     │
│ ● 所有人最后都后悔的故事      │
│   刚刚                       │
│ ○ 主角看见死亡时间            │
│   昨天                       │
│ ○ 没有标题的灵感              │
│   7月16日                    │
├─────────────────────────────┤
│ 我的小说                     │
│ 阅读与修改                   │
│ 故事记忆                     │
│ AI 任务                      │
│ 资料                         │
│ 设置                         │
└─────────────────────────────┘
```

规则：

- 当前对话必须明显高亮；
- 每条历史显示人话标题和最近时间；
- 新对话不会删除或覆盖旧对话；
- 切换对话时保存原对话草稿、滚动和关联项目；
- 功能入口位于历史列表之后，但不与某条对话混成项目树；
- 功能入口受真实能力和数据状态控制：没有真实页面、真实数据或有意义空状态时必须隐藏，不能放“以后开放”的死页面；
- S1 中“我的小说”显示真实项目或可行动的空状态，“AI 任务”显示真实任务/恢复状态，“设置”只显示已可用设置；“阅读与修改”仅在存在可读章节时显示，“故事记忆”仅在存在可解释内容时显示，“资料”在 S3 前隐藏；
- 到 S6 所有入口均已有真实能力后，才稳定呈现完整列表。

### 4.2 导航规则：功能页只在左侧内部 push

用户点击“我的小说”后，左侧区域进入自己的 `NavigationStack` 页面，出现返回按钮。不是在左侧原地展开项目树，也不替换中心对话。

```text
┌─────────────────────────────┐
│ ← 返回       我的小说         │
├─────────────────────────────┤
│ 正在创作                     │
│ 逆命登仙 · 准备第4章          │
│                              │
│ 其他小说                     │
│ 暂未命名的新书                │
│                              │
│ ＋ 创建新小说                 │
└─────────────────────────────┘

中心区域：仍是原对话、原滚动位置、原输入草稿和原任务状态
```

返回后恢复左侧默认页和原来的对话历史滚动位置。

### 4.3 我的小说

```text
┌─────────────────────────────┐
│ ← 导航       我的小说         │
├─────────────────────────────┤
│ 正在创作                     │
│ ┌─────────────────────────┐ │
│ │ 逆命登仙                 │ │
│ │ 前三章已确定 · 准备第4章 │ │
│ │ 今天 18:32               │ │
│ │ [继续创作] [打开小说]    │ │
│ └─────────────────────────┘ │
│                             │
│ 其他小说                     │
│ ┌─────────────────────────┐ │
│ │ 暂未命名的新书           │ │
│ │ 只保存了一个念头         │ │
│ └─────────────────────────┘ │
│                             │
│ ＋ 创建新小说                │
└─────────────────────────────┘
```

点击“创建新小说”后回到中心对话，由仓颉接手：

> 新小说已经建好了。你脑子里现在有什么？一句话、一个人物、一幅画面都可以。

### 4.4 阅读与修改

左侧“阅读与修改”页只负责选择项目和章节。点开正文后，以**全屏阅读层**覆盖当前驾驶舱，给小说足够宽度；中心对话仍在底层保持，不被销毁或重新创建。

- 项目和章节选择；
- 连续正文阅读；
- 搜索；
- 当前版本和是否已确定的人话状态；
- 章节目录默认收起；
- “和仓颉聊这章”关闭阅读层，回到原对话并带上章节上下文；
- 关闭阅读层后恢复原对话滚动、输入草稿、流式状态和左右导航位置；
- 阅读期间后台任务可以继续，但重大决定仍必须暂停等待用户。

### 4.5 故事记忆

普通用户看到：

- 这本书目前确定了什么；
- 主要人物现在是什么状态；
- 这个人物现在知道什么；
- 前面埋下、后面还要交代的事情；
- 时间和地点；
- 还没确定或存在矛盾的内容。

默认不要求用户手工维护。高级详情才展示正典、知识边界、Promise Ledger 和版本来源。

### 4.6 AI 任务

显示：

- 当前正在做什么；
- 已完成步骤；
- 下一步；
- 已花费用；
- 是否等待用户；
- 暂停、恢复和取消；
- 出错后的可恢复状态。

不展示内部思维链。可以展示输入来源、调用模型、检查结论和真实工具回执。

## 5. 右侧“这次聊出来的内容”

右侧不是第二套后台，而是当前对话的结果盒子。

### 5.1 信息顺序

```text
这次聊出来的内容

等你决定（2）
· 第一章是否现在暴露能力
· 是否接受本次局部修改

已经确定
· 主角被逐出宗门
· 暂时不立刻复仇

正在进行
· 第一章人物一致性检查

最近修改
· 开场改为雨夜
· 保留你标记的两句话

[查看更多]
```

### 5.2 右侧卡片规则

- 只展示与当前对话相关的最新结果；
- 用户确认后，“等你决定”卡片立即消失并移动到“已经确定”或“最近修改”；
- 不能出现审批成功但旧审批卡一直悬挂；
- 横屏和竖屏都必须能完整查看关键内容；
- 工程字段放入“技术详情”；
- 点击内容应打开对应正文、方向总结或任务，而不是跳到无关页面。

## 6. 对话中的仓颉

### 6.1 回答结构

仓颉的默认回复优先包含：

1. 对用户意思的简短理解；
2. 已经实际完成的动作；
3. 一个最重要的问题或下一步；
4. 必要时提供 2–3 个明显不同的选择；
5. 可展开的详情。

禁止每次都输出长报告。

### 6.2 动态追问示例

用户：

> 我想写一个所有人都觉得他是废物，但其实特别厉害的主角。

仓颉：

> 这个念头最关键的不是“他厉害”，而是他为什么一直没让别人知道。你更想看哪一种？
>
> - 他主动隐藏，别人欺负到头上才出手；
> - 他自己也不知道身体里藏着什么；
> - 他以前很强，现在失去一切重新爬起来；
> - 都不对，我脑子里是另一种画面。

用户：

> 不知道，就是觉得这种设定挺爽。

仓颉：

> 那先不讲设定。我写两个 150 字的小开场，你只要告诉我更想继续看哪个。

### 6.3 用户否定时

用户：

> 都不是这个感觉。

仓颉必须：

- 承认理解偏了；
- 总结已排除方向；
- 换一个维度；
- 不把刚否定的内容换词重提。

例如：

> 明白，问题不在“隐藏实力的原因”，是我问得太早了。你现在更像是只有一个“所有人以后都会后悔”的感觉。那我换个问法：你最想让哪类人后悔？

## 7. 正文阅读与自由批注

### 7.1 阅读页

```text
┌──────────────────────────────────────┐
│ ←  第一章 雨夜逐徒      章节目录  ··· │
├──────────────────────────────────────┤
│                                      │
│   雨水顺着残破的山门落下。             │
│                                      │
│   林越跪在青石阶前，没有抬头。          │
│   大殿里传来的每一句话，都像早已写好     │
│   的判词。                             │
│                                      │
│   “从今日起，你不再是青玄宗弟子。”      │
│                                      │
│   ……                                  │
│                                      │
├──────────────────────────────────────┤
│ 和仓颉聊这章      当前版本已自动保存     │
└──────────────────────────────────────┘
```

正文首先必须像小说，不能像标注数据集。

### 7.2 选区菜单

用户长按并拖动任意字句：

```text
复制 | 全选 | 喜欢 | 不喜欢 | 原样保留 | 更多
```

“更多”：

- 方向不对；
- 问仓颉；
- 补充一句。

### 7.3 六种动作

| 动作 | 用户理解 | 系统行为 |
|---|---|---|
| 喜欢 | 这种感觉以后可以多一点 | 保存正向偏好证据，不逐字锁死 |
| 不喜欢 | 这里不对，但我可能说不清 | 保存负向证据，原因可为空，启动诊断 |
| 原样保留 | 修改时这些字不能动 | 建立精确强保护范围 |
| 方向不对 | 不只是句子，整体走偏了 | 提升诊断范围，先做影响分析 |
| 问仓颉 | 我想问这段什么意思或为什么这样写 | 带选区和上下文回到对话 |
| 补充一句 | 我想直接加一句说明 | 保存用户备注，不自动当成正文 |

### 7.4 批注回到对话

```text
你标记了“不喜欢”：
“林越没有愤怒，只觉得这一切十分可笑。”

仓颉：
我看了前后内容。我先猜一下：
你不喜欢的可能不是这句话本身，而是林越在这里太冷静，
让前面的羞辱没有真正落到他身上。

[对，他应该更难受]
[不是，他可以冷静，但不能这么装]
[都不对，再换个问法]
[我自己说]
```

Agent 必须读取精确选区、前后文、章节版本、标签、备注以及同章其他喜欢/保护内容。

## 8. 修改前确认

```text
我大概找到问题了。

你想保留：
· 雨夜逐出宗门的开场；
· 师父冷漠宣判的感觉；
· 你标记为“原样保留”的两句话。

这次准备修改：
· 让林越表面克制，但身体反应暴露真实情绪；
· 删除那句过于故作轻松的内心话；
· 加强他和师父之间没有说出口的矛盾。

预计只改第一章中间约 700 字。
预计花费约 1～2 元。

[就这样改] [再聊聊] [取消]
```

默认不出现 Rewrite Scope、Artifact Revision、Context Hash、Binding 或 Receipt。

## 9. 从一个念头到小说的完整体验

### 9.1 一个念头

用户：

> 我想写一个主角能看见别人死亡倒计时的小说。

仓颉：

> 这个念头最有意思的地方，可能不是“看见死亡”，而是明知道一个人快死了，却不知道该不该救。我先确认一种感觉：你更想看主角利用能力改变命运，还是不断发现有些死亡根本改变不了？

### 9.2 用户说不知道

仓颉改用两个短开场，而不是要求用户给主线和爽点。

### 9.3 人话总结

```text
这本书目前确定的方向

大概故事：……
你想要的感觉：……
已经排除：……
还没确定：……

[对，就按这个继续]
[大体对，有地方要改]
[不对，重新理解]
[我也不知道，先试写看看]
```

### 9.4 创建项目和开始第一章

确认后由仓颉真实创建项目、保存方向、建立初始人物/规则/计划并估算预算。

```text
小说已经建好了，刚才确认的内容也保存好了。
第一章会直接出现死亡倒计时，但不会一次解释完原理。
预计 8～12 分钟，约 3～6 元。

[开始写第一章]
[先看看准备怎么写]
[预算再低一点]
```

### 9.5 生成期间

```text
正在写第一章

✓ 整理本章要发生的事
✓ 检查主角现在知道什么
● 正在生成正文
○ 检查人物有没有突然变样
○ 检查前后有没有矛盾
○ 保存可恢复进度
```

用户可以直接说暂停、继续、现在到哪了、还要多少钱。

### 9.6 阅读、反馈、修改

用户像读者一样阅读和批注。整章“不对味”也允许，不要求先给专业原因。仓颉诊断后说明改什么、不改什么和预计费用，再生成一个主版本和必要短备选。

### 9.7 确认章节

```text
第一章已经确定。
```

## 10. 术语翻译契约

| 内部术语 | 默认用户文案 |
|---|---|
| Creative Contract / Bible | 这本书目前确定的方向 |
| Canon | 已经确定、以后不能随便矛盾的设定 |
| Working Canon | AI 暂时按这个继续写的内容 |
| Artifact | 这次聊出来的内容 |
| Approval | 就按这个继续吗 |
| Frozen Chapter | 这章确定了 |
| Rewrite Scope | 这次准备改哪些地方 |
| Tool Receipt | 仓颉刚才实际做了什么 |
| Checkpoint | 已经保存的恢复位置 |
| Branch | 另一个版本 |
| Promise Ledger | 前面埋下、后面还要交代的事情 |
| Character Knowledge | 这个人物现在知道什么 |
| Reconciliation | 正在确认刚才是否真的完成 |

所有用户文案先写大白话；专业术语只在技术详情或高级设置中出现，并附解释。

## 11. 禁止的设计

1. 默认打开创建小说表单；
2. 一次问主线、爽点、人物弧、世界观和结局；
3. 固定三个问题无论回答什么都继续；
4. 不满意必须先填写原因或选择专业分类；
5. 正文每段一个框、一个 Lock 按钮；
6. 审批卡确认后不关闭；
7. 横屏用省略号隐藏关键审批内容；
8. 默认显示 Hash、Revision、Binding、Receipt；
9. Agent 只口头说执行，后台没有真实工具结果；
10. 让用户手动选择哪个 Agent、提示词、上下文或审校轮次；
11. 一次生成多个完整方案，把判断负担推给用户；
12. 用 Keychain、Candidate Set、Probe 等诊断页面代表产品主页；
13. 把“第二次覆盖安装”当成正常升级步骤；
14. AI 静默决定人物死亡、主线换轨、大范围重写或预算越界。

## 12. 设计定调验收

后续每一阶段都必须通过以下五个问题：

1. 一个只会看小说的人，不看说明能否知道下一步做什么？
2. 用户是否只需要说人话、看正文和表达感觉？
3. Agent 是否真的能操作软件，而不是只有聊天外观？
4. 专业复杂度是否仍然完整存在，但默认隐藏？
5. 用户真正重要的决定是否仍由用户掌握？

如果任一答案为“否”，即使技术功能已经完成，也不能称为产品体验完成。
